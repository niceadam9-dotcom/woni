# 소방계획서 탭 상세 설계 (4차·개정2) — 장(章) 기반 다단계 탭 구조

> 2026-07-21 개정. 구조 = **탭(소방계획서) > 목차(장/서식 좌측 트리) > 서식 화면(섹션 카드)** — §1 참조.
> 서식 번호·제목은 실제 양식(양식-placeholder.hwpx)에서 추출한 원문 그대로.
> 서식 1.4는 `erp_goal/image-1.png` 양식과 동일한 체크박스 화면으로 구현(§4).
> 유지 원칙: **기본정보 탭 유지 · 건물·시설 탭의 건물목록 유지**(소방시설현황 패널만 소방계획서로 이동)

## 0. 최종 목적

**고객관리에 입력·저장된 값을 기반으로, 한글 SDK 병합 파이프라인을 통해 두 문서를 자동 출력한다:**

1. **소방계획서** (소방청 표준양식 — HWP + PDF + 웹 미리보기)
2. **소방시설등 자체점검 실시결과 보고서** (별지 제9호서식 — 작동·종합/최초점검)

설계 전체가 이 목적에 종속된다:
- 이 문서의 **탭 구조(§1~§6)** = 두 문서의 공통 필수값을 한 번만 입력하는 화면 (겹치는 데이터가 별지 9호 1~3쪽의 약 80% — §9-2)
- **시설 코드 표준화(§4-3)** = 서식 1.4와 별지 9호 3쪽 체크리스트를 같은 코드 1벌로 병합
- **한글 SDK 파이프라인**(placeholder 심기 → 치환 병합 → HWP·PDF 2단계 등록, 검증 완료) = 두 문서 공용 출력 엔진
- **보고서 페이지(§10)** = 두 문서의 생성 진입점 (소방계획서 = 고객 선택, 별지 9호 = 점검 건 선택)
- 별지 9호 전용 추가분: 점검기간·점검자·점검인력(직원 자격정보), 점검결과(시설별 양호/불량) — 점검 도메인에서 입력·연동(§9-3)

## 1. 전체 탭 계층 (2026-07-21 개정 — 좌측 목차 트리 + 섹션 카드 채택)

계층 = **탭 > 목차(장/서식) > 화면(섹션 카드)** 3단계. 종전의 "장 가로 서브탭 + 서식 가로 탭 + 세부 탭" 4단계에서
① 장·서식 이동을 **좌측 목차 트리**로 통합(제안 A) ② 서식 내부 세부 탭을 **같은 화면의 섹션 카드 스크롤**로 대체(제안 B).

```
고객 상세 (상위 탭 — 기존 유지)
[기본정보] [건물·시설] [관계인] [소방계획서 ⚠] [청구·수금] [이력]
                              │
                              ▼ 소방계획서 탭 진입 = 빠른 입력 화면 (§1-1, 기본)
┌────────────────────────────────────────────────────────────┐
│ 준비율(필수) ▓▓▓▓░ · 누락 요약      [HWP 생성] [PDF 생성] [고급 ⇄] │ ← 생성 바
│  필수 공통값 체크리스트(대장 불러오기 → 빠진 것만 입력)              │
│  최근 보관함 1줄 요약 (2026년 · PDF ready · [열기])               │
└────────────────────────────────────────────────────────────┘
                              │ [고급] 전환 시 — 서식 전체 모드
┌──────────────┬─────────────────────────────────────────────┐
│ 개정이력·보관   │  준비율(전체) · 생성 바 (상단 고정)               │
│ ▾ 1장 안전관리  │ ┌ 서식 1.1 건축물 일반현황 ──────────────┐      │
│   1.1 일반현황✓ │ │ [시설현황 카드]                        │      │
│   1.2 세부현황  │ │ [운영현황 카드]   ← 세부 탭 아님,        │      │
│   1.3 위치·진입 │ │ [화재보험 카드]     같은 화면 세로 스크롤  │      │
│   1.4 소방시설✓ │ │                  (좌측 상단 앵커 점프)  │      │
│   …1.10 1.11  │ │                        [저장 1개]      │      │
│ ▸ 2장 자위소방대│ └───────────────────────────────────────┘      │
│ ▸ 3장 피난계획  │                                              │
└──────────────┴─────────────────────────────────────────────┘
```

- **좌측 목차 트리**: 문서 목차와 동일 모양(장 접기/펼치기) — 이동 1클릭, 현재 위치·완성도(✓/n/m/⚠) 상시 표시.
  모바일은 상단 드롭다운(장·서식 선택)으로 폴백
- **섹션 카드**: 서식 화면 안은 탭 없이 카드 세로 스크롤(카드 앵커 점프) — 저장 버튼은 서식당 1개
- URL: `/customers/{id}?tab=plan&form=1.1` (+`#카드앵커`) — 딥링크 유지(누락 안내 → 해당 서식 바로 이동)
- 뱃지: 목차 항목별 완성도 표시, 합산이 상위 소방계획서 탭 뱃지
- 저장: **서식 단위 저장 버튼 1개**, 미저장 이동 시 확인
- **기본 진입: 빠른 입력 화면(§1-1)** — 개정이력·보관은 고급 모드 목차의 첫 항목 (종전 "기본 진입=개정이력·보관" 규정은 폐기)

### 1-1. 빠른 입력 모드(기본) ↔ 서식 전체 모드(고급) — 2026-07-21 채택

4단계 탭은 서식 충실도용 **고급 모드**로 두고, 입력 동선의 기본은 **빠른 입력 모드**로 한다:

- **빠른 입력(기본 진입 확정)**: 두 문서(소방계획서 + 별지 9호) 생성에 필요한 **공통 필수값만** 모은 단일 스크롤 화면
  + **최근 보관함 1줄 요약**. 동선 = [건축물대장 불러오기(§11-1)] → 빠진 필드만 채움 → [생성]. 신규 고객도 5분 내 문서 출력이 목표
- **서식 전체(고급)**: 좌측 목차 트리 + 서식 화면(섹션 카드) — 세부를 다듬을 때 진입 (빠른 입력 화면 상단 토글로 전환)
- 준비율 게이지 이원화: **필수 완성도**(빠른 입력 기준 — 생성 가능 여부)와 **전체 완성도**(서식 전체) 분리 표시
- 필수 필드 정의 = 별지 9호 1~3쪽 + 소방계획서 준비율 어휘(fire-plan-readiness)의 합집합 — 구현 시 상수 1곳으로 관리

## 2. 서브탭 ① 개정이력·보관

소방계획서 문서 자체의 이력 관리 — "개정이력 + HWP 첨부 저장 + 생성 이력"을 한 화면에.

```
┌ 개정이력 ──────────────────────────────────────────┐
│ 순번  일자        주요 개정내용            작성자     │
│  1   2025-07-01  2025년 소방계획서 작성    홍길동    │
│  2   2026-07-21  2026년 소방계획서 작성    김직원    │ ← 생성 시 자동 누적
│ 이번 작성일 [2026-07-21] 개정 내용 [            ] 저장 │
├ 보관함 ───────────────────────────────────────────┤
│ ▸2026 2026년 소방계획서  [미리보기][인쇄][PDF][HWP][연차][✕]│
│ ▸2025 …  (PDF 변환 중 스피너 · 제출추적 · 부속자료 포함)   │
│ [+ 소방계획서 HWP/PDF 직접 업로드(외부 작성분 첨부 저장)]   │
└───────────────────────────────────────────────────┘
```

- 개정이력 = fire_plans 행 기반 자동 목록 + 이번 작성일·개정내용 입력(요구 0·1·2 — HWP 개정이력 표에 순번대로 병합 출력)
- 보관함 = 현행 fire-plans-client 이동(2단계 등록 표시 포함). 외부에서 작성한 HWP 첨부 업로드도 여기서.

### 4-3. 시설 코드 표준화 — "매치 안 됨"(요구 10)의 원인과 해법

현행 `fire_facilities.facility_code`는 **축약 22종**(점검 보고서용 CATALOG), 서식 1.4는 **정식 명칭 42종+피난기구 하위 8종**이라 불일치가 발생한다.

**해법: 서식 1.4의 정식 명칭 42종(+하위 8종)을 새 표준 코드로 채택**하고, 기존 데이터는 매핑 마이그레이션으로 이관.

| 현행 코드 (22종) | → 서식 1.4 표준 코드 | 비고 |
|---|---|---|
| 소화기구 | 소화기구 및 자동소화장치 | 1:1 |
| 옥내소화전 / 옥외소화전 | 옥내소화전설비 / 옥외소화전설비 | 1:1 |

| 상수도소화용수설비 / 소화수조·저수조 | 상수도소화용수설비 / 소화수조 및 저수조 | 1:1 |
| 제연설비 | 거실제연설비 + 부속실 등 제연설비 中 | **1:N — '거실제연설비'로 이관 후 재확인 뱃지** |
| 연결송수관/연결살수/비상콘센트/무선통신보조 | 동일 명칭(…설비) | 1:1 · 연소방지설비는 신규 |

- 마이그레이션(097 예정): 위 매핑으로 `fire_facilities.facility_code` UPDATE + 1:N 이관 행에 `detail.note='코드 이관 — 세부 확인 필요'` 표시

- 점검 보고서 등 기존 축약 코드 사용처는 grep 후 동시 수정(전환 기간 없이 일괄)

  §9-4 음성 확인 UX도 점검표 입력 화면에서), 3쪽 양호/불량은 설비 시트별 X 유무로 자동 계산

- 생성 버튼은 필수 항목(①②③) 충족 시 활성 — 미충족 항목은 경고색 + 딥링크로 유도

### 9-7. 별지 10·11호 — 불량 생애주기 관리 (2026-07-21 채택)

서식 실측(2026-07-21, `_form/` SDK 추출): 둘 다 1쪽 단순 서식 — 핵심은 병합이 아니라 **불량 데이터의 생애주기**.

| 서식 | 구성 | 시점 |
|---|---|---|
| 10호 이행계획서 | 대상물 정보 + 이행조치 계획(조치 사항·기간, 행 단위) + 전체 필요기간 | **9호와 함께 제출**(9호의 법정 첨부서류) |

  + action_done_at(완료일) + action_note + photo_before/after → 별지 11호 완료 행
점검표 X → 불량 자동 등록(기존) → 10호 계획 행 → 11호 완료 행 (같은 데이터의 생애주기)
```

- 대상물 정보 = 9호 공통값 재사용(추가 입력 0). 11호 공사업체 = company_profile 자동(외부 업체 시 수정 가능)
- 생성 = 동일 placeholder 파이프라인, 큐 `report_type='report10'/'report11'`, 저장 규약 §9-6⑤ 동일
- 템플릿 = `_form/` 별지 10·11호 (법제처 개정 감지 대상에 포함)

**입력 동선 — 준비 화면(§9-6⑦) 단계 추가, 새 페이지 없음**

│ ⑤ 이행조치 계획   ⚠ 불량 3건 중 2건 계획 미입력 [입력] │ ← 불량 있을 때만 표시
│    [별지 9호+10호 생성]  ← 불량 있으면 세트 생성(10호는 9호의 법정 첨부)

│ ⑥ 이행완료 처리   1/3 완료 · 전후 사진 2건 누락 [입력] │

```

- 불량 0건이면 ⑤⑥ 미표시 — 화면 복잡도 불변

기한·알림도 없음(보관 의무만). 특별점검용 코드·데이터는 유지, 조건부 표시로만 숨김.

### 11-4. 입력 컴포넌트 공통 규칙

- 숫자 필드는 단위 표시(㎡·kW·kVA·m·대·명) + 숫자 키패드(모바일 inputmode)
