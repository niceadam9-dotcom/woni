# 고객관리 탭 개편 + 건물관리 통합 설계 (2026-07-16 사용자 확정)

## 1. 목적·배경

고객 상세가 대형 카드 7~8개(기본정보→관계인→건물→소방시설→사업자·자동이체→소방계획서→점검 이력)를
항상 전부 펼친 세로 스크롤이라 **입력 위치를 찾기 어렵고 산만하다**. 건물관리 메뉴는 별도로 있으나
건물은 항상 고객 소속(customer_id 필수)이고 소방시설·층별은 이미 고객 상세에서 관리하므로
**두 메뉴를 오가는 동선이 중복**된다.

**확정 사항 (2026-07-16 사용자)**:
- 방안 3(탭 구조 전면 개편) 채택
- 건물관리는 **A안 — 메뉴 완전 삭제**(같은 날 B안→A안 재확정), 등록·수정·조회 모두 고객 상세 탭으로 일원화
- 탭 구성 6개 그대로: 기본정보 │ 건물·시설 │ 관계인 │ 소방계획서 │ 청구·수금 │ 이력

## 2. 고객 상세 — 탭 구조

```
[고객명 헤더 — 상태 · 담당 · 점검구분 · ◀ 이전 | 다음 ▶ · 빠른 액션]   ← 항상 표시 (탭 밖, §6-C-3)
──────────────────────────────────────────────────┬────────────
 기본정보 │ 건물·시설 ⚠ │ 관계인 (2) │ 소방계획서 6/9 │  │ 요약 패널
 청구·수금 ⚠ │ 이력 (점검 07-10)     ← 탭 상태 뱃지 (§4)  │ 주소·연락처
──────────────────────────────────────────────────│ 담당·계획일
 (선택된 탭 내용만 표시)                              │ 대표 ☎
                                                   │ 최근 점검
                                                   └ §6-C-2 (넓은 화면)
```

| 탭 | key | 내용 (기존 컴포넌트 이동 — 로직 변경 없음) |
|---|---|---|
| 기본정보 | `info` (기본값) | 기본정보 카드(edit-customer-info-client) + 담당직원 배정 |
| 건물·시설 | `buildings` | 건물 목록 + **건물 등록·수정 인라인 패널(§5)** + 소방시설 현황(FacilitiesClient, 층별 구역 포함) |
| 관계인 | `contacts` | 관계인 정보 카드 |
| 소방계획서 | `plan` | 계획서 정보 패널(FirePlanInfoPanel) + 표준양식 생성 + 보관함(FirePlansClient) |
| 청구·수금 | `billing` | 사업자정보 + 자동이체(BillingClient) |
| 이력 | `history` | 점검 이력 + 변경 이력 통합 타임라인 |

### 렌더링 방식 (구현 원칙)

- 페이지는 현행 **서버 컴포넌트 데이터 조회를 그대로 유지**하고, 클라이언트 탭 셸(`CustomerTabs`)이
  섹션 표시만 전환한다 (전 섹션 서버 렌더 → 탭별 show/hide). 데이터 페치 리팩토링 없음 = 리스크 최소.
- 탭 상태는 **URL 쿼리 `?tab=`** 으로 관리 (`router.replace`, scroll: false) — 새로고침·뒤로가기·딥링크 유지.
- 유효하지 않은 tab 값은 `info`로 폴백.

## 4. 탭 상태 뱃지 (입력현황 스트립 → 탭 뱃지로 통합, 2026-07-16 확정)

별도 스트립 없이 **탭 라벨에 상태를 병기**해 탭을 열지 않아도 판단이 끝나게 한다.
계획서 정보 패널의 준비율 패턴(2026-07-16 구현·검증 완료)을 페이지 레벨로 확장.

| 탭 | 뱃지 | 판정 기준 |
|---|---|---|
| 기본정보 | ⚠ (미완 시) | 점검구분·점검계획일·담당 배정 존재 |
| 건물·시설 | ⚠ (미완 시) | 활성 건물 ≥1 + 용도·연면적 입력 |
| 관계인 | (n) 인원수, 0이면 ⚠ | 대표 관계인 존재 |
| 소방계획서 | n/9 준비율 | **`src/lib/fire-plan-readiness.ts` computeFirePlanReadiness 재사용** |
| 청구·수금 | ⚠ (미완 시) | 사업자정보 등록 여부 |
| 이력 | 최근 점검일 (MM-DD) | — 상태성 정보 |

- 미완 ⚠은 앰버, 완료는 뱃지 없음(관계인 인원수·이력 날짜는 항상 표시) — 서버 컴포넌트에서 이미 조회한 데이터로 계산 (추가 쿼리 없음).

## 5. 건물 등록·수정 인라인 패널 (건물·시설 탭)

- `/buildings/new`·`/buildings/[id]` 페이지 이동 대신 **탭 안에서 패널 확장**으로 처리.
- 기존 `building-new-client.tsx`·`building-detail-client.tsx`의 폼 섹션을 **패널 컴포넌트로 재사용**
  — 저장 액션(createBuildingAction/updateBuildingAction) 변경 없음.
- 동작: 건물 목록 행 클릭 → 아래로 편집 패널 확장 / [+ 건물 등록] → 빈 패널 확장.
  저장·취소 시 패널 닫고 목록 갱신(router.refresh). URL `&b={bid}` 반영으로 딥링크 지원.
- 권한: 현행 유지 (등록·수정 manager/admin, 조회 전원).

### 5-A. 주소·건축물대장 중복 입력 제거 (2026-07-16 사용자 지적 반영)

**현행 중복 3곳 (확인됨)**:
| # | 중복 | 현행 코드 근거 |
|---|---|---|
| ① | 고객 등록 때 주소를 검색했는데 건물 등록 폼에서 **주소 재검색** 필요 (프리필은 도로명만, 지번·우편번호 없음) | building-new-client.tsx:85-134 |
| ② | bcode(법정동코드) 미저장 → 계획서 패널 [건축물대장에서 다시 가져오기]가 **매번 Daum 주소창 재확인** | fire-plan-info-panel.tsx refetchLedger |
| ③ | 건물 등록 폼에는 **대장 자동 조회가 없어** 용도·연면적·층수를 수기 재입력 (고객 등록 폼에만 존재) | fetchBuildingLedgerAction 참조처 = customer-new뿐 |

**해소 설계**:
1. **마이그레이션 092** — `buildings.bcode TEXT`(법정동코드 10자리)·`buildings.address_jibun TEXT` 신설.
   주소 검색이 일어나는 모든 지점(고객 등록·건물 패널)에서 Daum 결과의 bcode·지번주소를 함께 저장.
   무파괴(IF NOT EXISTS), 운영 적용은 사용자 승인 후.
2. **인라인 건물 패널 주소 상속** — 신규 건물 패널은 고객 주소·우편번호(+저장된 bcode·지번)를
   기본값으로 자동 상속 ("고객 주소와 동일" 체크, 기본 켜짐). 다른 주소일 때만 주소 검색 실행 → 중복 ① 제거.
3. **대장 조회 일원화·원클릭화** — 패널에 [건축물대장 조회] 버튼: 저장된 bcode가 있으면 **주소창 없이 즉시 조회**,
   없으면 주소창 1회 확인 후 bcode를 백필 저장(다음부터 원클릭). 계획서 정보 패널의
   [건축물대장에서 다시 가져오기]도 같은 로직을 재사용하도록 개선 → 중복 ② 제거.
4. **패널에 대장 자동 조회 포함** — bcode 확보 시(상속 또는 검색) 대장을 자동 조회해 용도·연면적·층수·
   구조·지붕·높이를 채움(기존 값은 덮어쓰지 않음, 고객 등록 폼과 동일 규칙) → 중복 ③ 제거.
5. **고객 신규 등록 폼 소폭 수정** — 주소 검색 시 받은 bcode·지번주소를 건물 생성 페이로드에 포함해 저장
   (createCustomerAction 건물 insert 확장. 신규 등록 폼 2단계화 자체는 여전히 후속 범위).

## 6-B. 고객 목록(첫 페이지) 개선 — 필수 컬럼 + 상세 접근 (2026-07-16 사용자 확정)

### A. 컬럼 필수화 (기본 6컬럼)

```
고객명(주소·건물 요약) │ 점검유형 │ 점검계획일 │ 담당직원 │ 상태 │ 바로가기
케이엔몰               │ 종합(연2회) │ 2026-09-10 │ 홍길동 │ 활성 │ 🏢 📄 ›
```

- **점검횟수 컬럼 삭제** — 점검유형 뱃지에 "종합(연2회)" 병기 (TYPE_ANNUAL 파생값 흡수)
- **계약일·사용승인일 기본 숨김** — [전체 컬럼] 토글(URL `?cols=full`)로 표시, 인라인 편집은 펼쳤을 때 그대로 동작
- 기본 보기 = 고객명·유형·계획일·담당·상태 5개 + 바로가기 (일상 업무 필수값만)

### B. 상세조회 접근 개선

1. **행 전체 클릭 = 상세 이동** — 인라인 편집 셀은 클릭 버블링 차단으로 공존, hover 행 강조 + pointer 커서
2. **고객명 = 상세 링크로 환원** — 이름 인라인 편집은 hover 시 연필 아이콘(✎)으로 분리 (실수 편집 방지)
3. **탭 딥링크 바로가기** — 행 끝 미니 아이콘 🏢(`?tab=buildings`)·📄(`?tab=plan`), 입력 누락 고객은 ⚠ 점 표시 (§3·§4와 결합)
4. **검색 자동완성 직행** — 검색창에서 고객명 제안 선택 시 목록 필터 대신 해당 고객 상세로 바로 이동 (customer-search-box)

   | 수신기위치 | 자주 쓰는 후보(1층 관리실/방재실/경비실) |

2. **읽기 요약 모드** — 패널 기본을 컴팩트 요약 뷰(2~3열 정의 리스트)로, 누락 항목은 앰버 "미입력" 표시.

1. **설치 시설 중심 편집** — 편집 뷰를 '설치 시설 (n종)' 목록으로 압축, [+ 시설 추가]는 검색 자동완성

3. **층별 합계 → 상세 자동 제안** — 소화기·유도등·비상조명 층별 합계를 해당 시설 상세가 비었을 때
   "층별 합계 n 적용" 버튼으로 제안 (중복 입력 제거)
4. **빠른 시작 배너** — 시설 0·층 0 건물에서 편집 진입 시 "기본 세트 + 층 자동 생성" 원클릭
5. (조사 후 판단) 최근 점검표에서 설치 시설 역추출 — 점검표 항목↔시설 코드 매핑 확인 선행 (follow_up)

### §12-A. 수량·상세 입력 개선 (2026-07-16 사용자 확정 A+B+C, T12)

6. **종류+수량 칩 편집기(A)** — 상세 자유 텍스트를 시설별 종류 프리셋 칩([분말 −12+ ✕] 등) + 수량
   스테퍼로 편집. **저장은 기존 문자열("분말 12, CO2 2") 직렬화 — DB·보고서 무변경.**
   기존 문구가 "이름 숫자, …" 패턴이면 칩으로 역파싱, 아니면 자유 텍스트 폴백(전환 버튼 제공).
   종류 프리셋은 facility-presets.ts DETAIL_TYPE_PRESETS (직접 입력 추가 허용)
7. **층별 합계 연동 확장(B)** — 감지기류(차동+연기+정온 합계) → 자동화재탐지설비 "감지기 nn" 제안 추가
   (기존 소화기·유도등·비상조명 3종 → 4종)
8. **층별 합계 불일치 경고(C)** — 칩 수량 합(자탐은 '감지기' 칩만)과 층별 합계가 다르면 앰버
   "층별 합계(n)와 불일치" 표시 — 계산만, 자동 수정 없음

- E2E·업무 습관: 건물 관리 메뉴·건물 상세 URL이 사라짐 → 전 경로 리다이렉트로 흡수, 메뉴 삭제는 사용자 공지 필요 — 낮음
- **기존 건물은 bcode가 비어 있음** → 대장 조회 시 주소창 1회 확인 후 백필 저장(§5-A-3)으로 자연 해소,
  일괄 백필 배치는 불필요 — 낮음
- **급수 자동 판정은 법정 기준 오판 리스크** → '제안+하이라이트+근거 툴팁'으로만 동작(자동 저장 없음),
  구현 전 시행령 별표 4 기준 확인 필수 — 중간(운영 절차로 관리)
- 신규 등록(customers/new) 폼은 이번 범위 아님 — "최소 등록 → 상세 보완" 2단계화는 후속(방안 1, 별도 지시 대기)
- status 관리: 구조 JSON(고객관리-탭개편-설계.json)은 Json_Rule.md 규약(criteria 단위·verified 증거·독립 검증) 적용
