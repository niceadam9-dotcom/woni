## 3. 기능 테스트 시나리오

각 케이스 형식: **ID / 사전조건 / 테스트 단계 / 기대 결과**. 결과는 §6 양식에 기록.

---

### A. 고객 등록 (Victory9 고객등록 §1~19, V10 §6)

#### A-1. 주소검색 최상단 배치 + 괄호 건물명 자동추출

- **사전조건**: admin 로그인, 고객관리 > 신규 등록 화면
- **단계**:
  1. 등록 폼에서 필드 순서 확인 — 주소 검색이 폼 최상단, 그 다음 건물명(고객명)인지 확인
  2. 주소 검색에 `양평로 92` 입력 → 결과에서 `경기 양평군 양평읍 양평로 92 (양평동우체국)` 선택
  3. 건물명 입력란 확인
- **기대 결과**:
  - 건물명에 `양평동우체국` 자동 입력됨 (괄호 안 텍스트 추출)
  - 입력란 옆 `[×]` 지우기 버튼 표시, 클릭 없이 즉시 편집 가능 (전체 선택 상태 → 바로 덮어쓰기 가능)

#### A-2. 괄호 없는 주소 → 빈칸 자동 포커스

- **단계**: 주소 검색에서 `경기 양평군 용문면 연수리 12` 처럼 괄호 없는 주소 선택
- **기대 결과**: 건물명 입력란이 빈 상태로 자동 포커스(커서 위치) → 클릭 없이 바로 타이핑 가능

#### A-3. 사용자 수정값 우선 저장

- **단계**: A-1에서 자동 추출된 `양평동우체국`을 `테스트종합빌딩`으로 수정 후 저장
- **기대 결과**: 수정값 `테스트종합빌딩`으로 저장됨 (자동추출값이 덮어쓰지 않음)

#### A-4. 지역 자동 입력 + 기본 지역 pre-fill

- **단계**:
  1. 신규 등록 화면 진입 직후 지역 필드 확인
  2. 주소 선택 후 지역(시/도, 시/군/구, 읍/면/동) 필드 확인
- **기대 결과**:
  - 진입 시 시/도=`경기도`, 시/군/구=`양평군` 기본값 pre-fill (company_profile 기본 지역)
  - 주소 선택 시 region_si / region_myeon / region_ri 가 주소에서 파싱되어 자동 입력
  - (재등록 시) 직전 등록에 사용한 읍/면이 localStorage 기반으로 우선 적용

#### A-5. 점검유형 라디오 필수 선택

- **단계**:
  1. 점검유형 UI 확인 — 대분류 라디오(소방안전관리/일반관리)
  2. `소방안전관리` 선택 → 중분류 라디오(종합/작동) 표시 확인
  3. 대분류 또는 중분류 미선택 상태로 저장 시도
- **기대 결과**:
  - 대분류 필수. 소방안전관리 선택 시 중분류(종합/작동)도 필수
  - `일반관리` 선택 시 중분류 라디오 미표시(또는 비활성)
  - 미선택 저장 시 유효성 오류 표시, 저장 차단
  - 구 명칭 `최초`, `기타`가 UI 어디에도 노출되지 않음 (→ `작동`으로 표시)

#### A-6. 관계인 — 대표 1명 필수 + 추가

- **단계**:
  1. 관계인 없이 저장 시도
  2. 대표 1명 입력 후 `추가` 로 직원 관계인 1명 더 입력 후 저장
- **기대 결과**: 대표 미입력 시 저장 차단. 대표 + 추가 관계인 정상 저장, 상세보기에서 확인 가능

#### A-7. 엔터키 저장 / 저장·수정 단일 폼

- **단계**:
  1. 모든 필수 항목 입력 후 마지막 입력란에서 Enter
  2. 저장된 고객을 다시 열어 같은 폼에서 값 수정 후 저장
- **기대 결과**: Enter로 저장 완료. 등록 폼과 수정 폼이 동일(단일 폼) — 수정사항 정상 반영

#### A-8. 고객코드 자동생성 + UI 숨김

- **단계**:
  1. 고객 2명 연속 등록
  2. 목록/상세 화면에서 고객코드 노출 여부 확인
  3. (DB 확인) `customers.customer_code` 값 비교
- **기대 결과**:
  - 고객코드 입력란 없음 — 시스템이 최종번호+1로 자동 생성
  - 목록/상세 UI에 고객코드 미노출 (내부 코드)

#### A-9. 화면 스크롤 없이 전체 표시

- **단계**: 1920×1080 해상도에서 등록 폼 전체 확인
- **기대 결과**: 세로 스크롤 없이 폼 전체가 한 화면에 표시

#### A-10. 건물관리 자동 동기화 (주소 2번 검색 불필요)

- **단계**:
  1. 고객 등록 시 입력한 주소로 저장
  2. 건물관리(또는 고객관리 내 건물 정보)에서 해당 고객의 건물 확인
  3. 건물 정보(용도/연면적/층수/준공/상태) 입력 → 고객 상세에서 동일 값 표시 확인
- **기대 결과**:
  - 고객 등록 주소가 건물 정보에 자동 반영 — 건물 등록 시 주소 재검색 불필요
  - 건물 정보 수정 시 고객 등록/상세 화면에도 동일 값 표시 (양방향 동기화)

#### A-11. 점검이력 자동 기록 (등록/변경/삭제)

- **단계**:
  1. 신규 고객 저장 → 상세보기 > 점검이력 확인
  2. 고객명 / 점검유형 / 계약일 / 사용승인일 / 주소 / 담당직원 / 관계인 을 각각 1회씩 수정
  3. 각 수정 후 점검이력 확인
- **기대 결과**:
  - 저장/변경/삭제 각각 이력 1건 추가
  - 이력에 변경 필드·변경 전후 값·변경자·일시 확인 가능
  - 고객등록 화면 이력과 고객관리 > 상세보기 > 점검이력이 동일 (Victory9 고객등록 §13)

### B. 고객관리 목록 / 통합 검색 (Victory9 고객관리 §3~5, V9-18, V10 §6)

#### B-1. 최신 등록 고객 최상단 정렬

- **단계**: 신규 고객 등록 직후 목록 조회
- **기대 결과**: 방금 등록한 고객이 목록 1행에 표시

#### B-2. 인라인 빠른 수정 (6개 필드)

- **사전조건**: T1 고객 존재
- **단계**: 목록에서 T1 행의 각 필드를 직접 클릭하여 수정 — ① 고객명(건물명) ② 점검유형 ③ 계약일 ④ 사용승인일 ⑤ 담당직원 ⑥ 상태
- **기대 결과**:
  - 6개 필드 모두 상세 진입 없이 목록에서 즉시 수정 가능
  - 수정 즉시 저장 + 화면 반영
  - ①·⑤ 수정 시 점검이력 기록 / ④ 수정 시 E-1 리셋 로직 동작(별도 검증)

#### B-3. 활성 ↔ 비활성 즉시 전환

- **단계**: 목록에서 비활성 버튼 클릭 → 상태 변경 확인 → 다시 활성 전환
- **기대 결과**: 목록 화면 내에서 즉시 전환. 비활성 고객은 점검달력에 회색 취소선으로 표시(F-6 연계)

#### B-4. 삭제 버튼

- **단계**: 임시 고객 1건 생성 후 목록에서 삭제
- **기대 결과**: 삭제 확인 절차 후 목록에서 제거. 관련 계획 항목은 달력에서 회색 취소선 처리(F-6). 점검이력에 삭제 기록

#### B-5. 통합 검색 — 스마트 자동감지

- **사전조건**: 시드 데이터(양평 고객) 존재
- **단계**: 단일 검색창에 아래를 순서대로 입력하여 각각 결과 확인
  | 입력 | 기대 검색 대상 |
  |------|---------------|
  | `빌딩` | 건물명(customer_name) 매칭 |
  | `경기` | region_si 매칭 |
  | `양평읍` | region_myeon 매칭 |
  | `양평로 92` | 도로명 주소(address) 매칭 |
  | `박개발` | 담당자(profiles.name) 매칭 |
  | `양평 테스트종합` | 주소 AND 건물명 복합 매칭 |
- **기대 결과**: 각 패턴이 자동 감지되어 해당 컬럼 기준 결과 반환. 매칭 키워드는 결과에서 **볼드 하이라이트**

#### B-6. 자동완성 드롭다운

- **단계**: 검색창에 `양평` 타이핑 (검색 실행 전)
- **기대 결과**: 드롭다운에 📍주소 / 🏢건물명 / 👤담당자 섹션 분리 표시 + 각 항목 건수 표시. 항목 클릭 시 해당 필터 즉시 적용

#### B-7. 빠른 필터 칩 + 담당자/지역 필터

- **단계**:
  1. `[전체] [종합] [작동] [일반관리]` 칩을 순서대로 클릭

- **기대 결과**: 칩 클릭 즉시 해당 읍/면 필터. 검색 결과에서 양평군 건이 상단 우선 표시

### D. 월간점검계획확정 — 1단계 점검일자확정 + 6단계 자동계산 (V9-9/10, V10 §5)

- **기대 결과**: `계획일(자동) 07-30 (예상)` / `확정일 — (미확정)` / 상태 `○ 계획중` 형태로 두 날짜가 구분 표시. 확정 후에는 차이(예: `-2일`)와 `✅ 확정` 표시

#### D-4. 1단계 점검일자확정 — 달력 팝업

- **단계**:
  1. T1 7월 항목의 `점검일자확정`(또는 미확정 날짜 셀) 클릭
  2. 커스텀 달력 팝업 확인 — 일요일(빨강)/토요일(파랑)/공휴일(빨강)/오늘(보라 테두리) 강조
  3. `2026-07-28` 클릭
- **기대 결과**:
  - 클릭 즉시 저장: `scheduled_date=2026-07-28`, `status='confirmed'`
  - `planned_date(07-30)`는 변경되지 않음
  - **step1~6_date 자동 계산 저장**: step1=07-28, step2=08-04(+5영업일), step3=08-11(+10영업일), step4=08-18(+15영업일), step5=08-28(step4+10절대일), step6=09-10(step5+10영업일) ※ 공휴일 개입 시 화면 계산값과 §5 규칙 대조

#### D-5. 확정 시 SMS 발송 + 발송 이력

- **기대 결과**: 발신번호(설정값)와 수신번호 자동 세팅. 발송 이력(`sms_sent_at`, `sms_recipients`, 내용) 확인 가능. 솔라피 미설정 환경이면 오류 대신 명확한 안내 메시지

#### D-6. 확정일 변경(재확정) → 6단계 재계산

- **기대 결과**: scheduled_date 갱신 + step1~6_date 전체 재계산. 달력/모니터링에도 즉시 반영. 이력 기록 추가

#### D-7. 확정 취소 → planned 리셋

#### D-8. 연/월 빠른 선택 + 월 이동

### E. 사용승인일 변경 → 리셋 (V9-5/7/18, V10 §5)

#### O-2. 최신 등록 최상위 (ADD-14)

- **단계**: 점검현황모니터링·점검업무에서 첫 행 확인 (최근 생성 항목과 대조)
- **기대 결과**: 두 화면 모두 created_at 내림차순

#### O-3. 점검업무 상태 필터 (ADD-15)

- **단계**: 상태 필터를 순서대로 적용 — 비완료(예정·진행중) / 완료 / 취소(비활성/삭제) / 세분 상태(예정/진행중/기한초과)
- **기대 결과**: 비완료에 완료·비활성 건 미포함, 취소에 비활성 고객 건 조회

#### O-4. 건물등록 주소 인계 (ADD-7)

- **단계**: 고객 상세 → [건물 등록](customer_id 파라미터) 진입 / 또는 건물등록에서 고객 선택
- **기대 결과**: 고객 주소·건물명(고객명) 자동 입력 — 주소 재검색 불필요. 기존 입력값이 있으면 덮어쓰지 않음

## 4. 전체 프로세스(E2E) 테스트 시나리오

| 5 | 확정 직후 SMS 발송 | 관계인 번호 자동 조회, 발송 이력 기록 (D-5) |
| 6 | 점검달력 재확인 | 점선→실선 전환 + 6단계 날짜 칩 표시, 긴박도 색상 (F-1/F-2) |
| 7 | park.dev(담당자)로 로그인 → 슬라이드 패널에서 1단계 완료 확인 | 모니터링 `점검일 (1단계)` 자동 기입 (G-2) |

| 12 | T1 점검이력 전체 검토 | 등록→확정→SMS→각 단계 완료가 시간순으로 모두 기록 |

### E2E-2. 작동 고객 연 1회 특별점검

1. T2(작동, 사용승인일 2026-06-30) 기준 — 특별점검이 **사용승인월 1회만** 생성되고 나머지 11개월은 monthly인지 확인 (C-2)
2. 특별 항목 확정 → 6단계 계산·진행이 종합과 동일 구조로 동작
3. 정기(monthly) 항목은 확정(실제점검일 업데이트)만 가능하고 **6단계가 생성되지 않는지** 확인
4. UI에 차수 대신 `연1회` 뱃지(emerald) 표시 확인

### E2E-3. 일반관리 이벤트 점검

1. T3에 자동 생성 계획이 없는지 확인 (C-2)
2. I-1 → I-2 순서로 등록→달력반영→확정→완료→이력 사이클 수행
3. 전 과정에서 6단계 미생성·연회색 표시 규칙 유지 확인

### E2E-4. 사용승인일 변경 리셋 사이클

1. T4 7월 항목 확정(step1~6 계산 완료)까지 진행
2. 사용승인일 2026-06-30 → 2026-07-31 변경 → E-1 전체 검증 (리셋 + planned_date 재계산 + 이력)
3. 새 planned_date 기준으로 재확정 → 6단계 재계산 → 정상 재개 (E-2)
4. 달력·점검업무·모니터링 세 화면 모두 "계획" 상태로 되돌아갔는지 교차 확인

### E2E-5. 담당자 변경 전파

1. T1 담당자 박개발 → 정개발 변경 (J-1)
2. 5개 화면(월간계획/점검업무/달력/지역배정/이력) 전파 확인
3. 두 employee 계정으로 각각 로그인하여 본인 목록 스코핑 확인 (D-9)
4. 지역별담당배정에서 일괄 배정으로 원복 → 동일 전파 확인 (J-2)

### E2E-6. 지연 발생 → 경보 사이클

1. 과거 날짜로 확정된 항목을 만들어 특정 step 마감일을 초과시킴
2. 달력: bg-red-600 지연 표시(취소선 없음), 요약 카드 🔴지연 +1, 고정 배너 노출 (F-2~F-4)
3. 사이드바 🔴 뱃지 건수 증가 (F-5)
4. cron 수동 호출 시 지연/D-Day 대상 포함 확인 (K-3)
5. 해당 단계 완료 처리 → 지연 표시 해제·건수 감소 확인

### E2E-7. 외부 테스터 시나리오 (3자 테스트)

1. ngrok 주소(또는 Vercel)로 외부 접속 → 로그인 (테스트 계정 전달)
2. employee 계정으로 E2E-1의 7~10단계(담당자 관점 업무) 수행
3. 모바일 브라우저에서 달력/슬라이드 패널 조작성 확인
4. 관리자 계정으로 admin 메뉴 접근, employee 계정으로 admin 메뉴 접근 차단 확인

## 5. 6단계 날짜 계산 검증 기준표

> **기준: 사용승인일(또는 1단계 확정일) 2026-06-30** — Victory9 §10 수정 반영(5단계=08-01)

| 단계 | 기준 | 계산 | 기대 마감일 | 요일 |
|------|------|------|------------|------|
| 1단계 점검일자확정 | 사용승인일 | +1 영업일 | 2026-07-01 | 수 |
| 2단계 배치확인서 보고서 작성 | 1단계 | +5 영업일 | 2026-07-08 | 수 |
| 3단계 관계인 보고서 제출 | 1단계 | +10 영업일 | 2026-07-15 | 수 |
| 4단계 소방서 보고서 제출·이행계획서 등록 | 1단계 | +15 영업일 | 2026-07-23 | 목 |
| 5단계 소방보수 완료 | 4단계 | **+10 절대일** (주말/공휴일 포함) | **2026-08-01** | 토 |
| 6단계 이행완료보고서 제출 | 5단계 | +10 영업일 | 2026-08-14 | 금 |

검증 방법:
1. E2E-1 4단계에서 1단계 확정일을 정확히 `2026-07-01`로 선택
2. 슬라이드 패널/모니터링/달력 3곳의 step2~6 날짜가 위 표와 모두 일치하는지 확인
3. 5단계가 토요일(08-01)로 나오는지 반드시 확인 — 절대일 계산이 영업일로 잘못 구현되면 08-04(화) 등으로 밀림
4. 공휴일이 낀 구간(예: 8월 광복절 주간)의 6단계 날짜도 별도 케이스로 1건 검증

### 기록 규칙

- **Fail 시**: 재현 단계 번호 + 기대/실제 결과 + 스크린샷을 비고에 남긴다.
- **Blocked 시**: 선행 케이스 ID를 명시한다 (예: "D-4 Fail로 진행 불가").
- 우선순위: **E2E-1 → E-1(리셋) → §5 날짜 검증 → 달력 긴박도(F-2)** 가 핵심 비즈니스 로직이므로 최우선 수행.
- 추가 요구(Victory10_add) 회귀 테스트 시: **E2E-8(등록 풀플로우) → M절(비활성 연동/자동취소·복원) → N-2/N-3(건축물대장)** 순 권장.
- 데이터 정리: 테스트 종료 후 T1~T4 및 파생 plan_item/이력은 삭제하거나 비활성 처리한다 (삭제 시 B-4·F-6 검증을 겸할 것).
