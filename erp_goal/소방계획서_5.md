## 2. 결정·확정 사항 (사용자 확인 완료)

1. **역할 분담**: 완전 통합 대신 — **보고서 센터 = 확인·생성·다운로드 단일 창구 / 타임라인 = 절차 진행 창구**(D-day·발송·제출 기록·업로드). 저장소는 이미 공유되므로 어느 쪽에서 만들어도 양쪽에 동일 표시.
2. **업로드 3종은 유지**: 배치확인서(협회 발급)·수리 계약서(체결 원본)·전/후 사진(현장 촬영)은 ERP가 생성할 수 없는 외부 발급/원본 문서 — 업로드가 본질. 대신 조회를 한곳에 모으고 누락을 감시한다.
3. **용어는 실무·법정 용어로 통일** (사용자: "실제사용용어 통일"). §3-5 용어표 적용, 라벨은 용어 사전 상수 1곳에서만 가져다 쓴다.
4. **자체점검 타임라인은 ①~⑥ 상시 표시** (사용자: "1단계~6단계"). 불량 0건이면 ⑤⑥을 숨기지 않고 흐리게 "해당없음 — 불량 0건 (불량 등록 시 활성화)".
5. **9-4 음성 입력은 보류**(녹음장비 미보유), 입력 기반(빠른 결과 입력 A안)으로 전환 완료 — 본 계획과 별개로 기완료.

## 4. 구현 범위 ①~⑪ 상세

### 4-0. 공통 UX 규약 (2026-07-26 재검토 — 모든 항목에 적용)

인지·사용 편의를 위한 전 화면 공통 규칙:

1. **상태는 색으로 즉시 인지** — 문서·단계 상태 색 규약 통일:
   ✅ 초록=보유·완료 / ⚠ 앰버=필요한데 없음(행동 필요) / 회색 흐림=해당없음(행동 불필요) / 빨강=기한 임박·초과.
   "회색은 무시해도 된다, 앰버만 처리하면 된다"가 한눈에 서게.
2. **2클릭 목표 (3클릭 상한)** — 어떤 문서든 확인·생성·다운로드·**업로드**가 2클릭 이내(달성 수단은 4-0-13 클릭 다이어트). **업로드도 그 자리에서**: 문서 행의 [업로드]는 파일 선택을 즉시 열고 같은 슬롯에 저장(타임라인과 저장소 공유라 어디서 올려도 동일), 드래그&드롭 지원. 화면 이동이 꼭 필요한 것(발송·제출 기록 등 절차)만 "→"를 붙여 이동임을 예고.
3. **버튼 문구 동사형 통일**: [바로 생성] [받기] [미리보기] [업로드 →] [사진 보기] [신고 정보 복사]. 같은 동작=같은 문구(화면마다 다른 표현 금지).
4. **행동 후 피드백 고정 위치**: 생성 요청·복사·업로드 성공은 해당 행 바로 아래 초록 한 줄(다른 곳을 볼 필요 없게), 실패는 빨강 + 원인 + 해결 딥링크.
5. **비어 있음 안내**: 목록이 비면 이유와 다음 행동을 함께 ("자체점검 건이 없습니다 — 점검 일정은 점검계획에서 확정하세요").
6. **모바일**: 현황 행 flex-wrap, 갤러리 반응형 그리드, 버튼 최소 높이 유지 — 현장 폰 사용 전제.
7. **상호 진입점**: 사이드바 "보고서" 외에 고객 상세 소방계획서 탭·점검 상세 타임라인에 "보고서 센터에서 보기 →" 역링크 — 어느 화면에서 시작해도 서로 오갈 수 있게.
8. **전체 직감 지도 — "어디서 무엇을 하는가" 한 문장씩** (모든 설계·안내 문구가 이 지도를 따른다):
   - **문서를 확인·생성·다운로드한다 → 보고서 센터** (고객명 검색 한 번)
   - **절차를 진행한다(기한·발송·제출 기록·업로드) → 점검 상세 타임라인** (①~⑥ 순서대로)
   - **값을 입력한다 → 고객 탭 서식 / 점검표** (문서는 입력의 결과일 뿐)
   - **사진은 불량내역에서 올리고, 갤러리에서 모아 보고, 비어 있으면 그 자리에서 채운다**
   이 네 문장이 성립하지 않는 배치는 채택하지 않는다. 신규 사용자 안내(각 화면 상단 캡션)도 이 문장을 그대로 쓴다.
9. **주 행동 버튼은 화면(구역)당 1개** — 지금 눌러야 할 다음 행동 하나만 보라색 primary, 나머지는 전부 테두리 보조 버튼. "뭘 눌러야 하지?"가 생기지 않게.
10. **하루의 시작점과 연결** — 사용자는 보고서 센터를 기억해서 찾아오지 않는다. **대시보드에 "문서 할 일" 위젯**(제출 기한 임박 n건 · 배치확인서 누락 n건 → 클릭 시 보고서 센터)을 두어, 시스템이 먼저 알려주고 사용자는 따라가기만 한다. 기존 알림(크론)과 같은 판정 로직 공유.

### 4-0-11. 입력 3원칙 — "쉽고 빠르게 입력" (2026-07-26 재검토 추가)

모든 입력 화면은 아래 우선순위를 따른다. **손 입력은 최후 수단**이다:

| 순위 | 원칙 | 이미 있는 수단 (재사용) | 이번 계획의 보강 |
|---|---|---|---|
| 1 | **자동으로 채울 수 있으면 자동** | 건축물대장 불러오기(미리보기→확정), 프리셋(주택/상가/공장·표준 문구), 전년도 이어받기, 점검시기 자동값 | 누락 칩→대장 즉시 실행(기완료), 배치신고 도우미(⑥ — 신고값 자동 구성) |
| 2 | **반복 입력은 일괄로** | 설치 설비 전체 양호(기완료), 소방계획서 다중 생성 | 갤러리 빈 슬롯 즉시 업로드(⑤), 바로 생성(②③) |
| 3 | **손 입력은 예외만** | 불량 태깅(검색→X+메모), 불량 사진, 미비 필드 | 준비 팝오버의 부족 항목 딥링크(② — 예외 입력처로 직행) |

- 배치 규칙: **자동 채움 버튼이 입력칸보다 위/먼저** 보이게 (사용자가 손으로 다 치고 나서 자동 버튼을 발견하는 일 방지).
- 입력 후 확인: 자동 채움 값은 앰버 하이라이트 → [확정 저장] (기존 대장 미리보기 패턴을 표준으로).

### 4-0-12. 모니터링 3층 구조 — "쉽게 감시" (2026-07-26 재검토 추가)

감시는 층마다 답하는 질문이 다르다. 층을 섞지 않는다:

| 층 | 화면 | 답하는 질문 | 보는 방법 |
|---|---|---|---|
| 1층 | **대시보드 "문서 할 일" 위젯**(4-0-10) | "오늘 내가 처리할 게 있나?" | 숫자 뱃지 2~3개, 클릭 시 해당 층으로 |
| 2층 | **보고서 센터 제출 현황판**(7-A A) | "문서·기한이 어디서 밀렸나?" | 상단 **숫자 요약 스트립** + 표에서 앰버·빨강만 훑기 |
| 3층 | **점검현황 모니터링**(기존) | "일정·현장 진행은 어떤가?" | 현행 유지 + 색 규약 정렬 |

- **숫자 요약 스트립** (2층 상단): `이번 달 자체점검 12 · 완료 8 · 9호 미제출 3 · 기한 초과 1 · 배치확인서 누락 2` — 각 숫자는 클릭하면 그 조건으로 표가 필터됨(숫자=필터 버튼). 관리자는 **숫자 5개만 보면 감시 끝**.
- 정렬 기본값 = 위험순(기한 초과 → 임박 D-day → 누락 → 정상) — 스크롤 없이 문제가 맨 위.
- 푸시 감시(기존 재사용): 기한 D-7/D-3/당일/경과 알림 크론 — 화면을 안 열어도 알림이 옴. 판정 로직은 화면과 크론이 1곳 공유(이중 판정 금지).
- 감시 동선 목표: **"대시보드 숫자 확인 → 클릭 → 앰버 행 처리" 3동작, 30초 안에 아침 점검 끝.**

### 4-0-13. 클릭 다이어트 — "2클릭 이내" (2026-07-26 사용자 지시: 클릭·입력 최소, 자동완성)

3클릭 원칙(4-0-2)을 상향: **목표 2클릭 이내, 3클릭은 상한**. 달성 수단 6가지:

**(1) 행동 자동완성 검색 — 검색 결과가 곧 실행 버튼 (핵심)**
보고서 센터 검색창(⓪)의 자동완성이 고객만 찾지 않고 **그 고객의 문서·행동을 바로 후보로** 띄운다:
```
[ 서림 ]  ← 2자 입력 (초성 "ㅅㄹ"도 인식)
  ▾ 서림사 — 문서 현황 열기
  ▾ 서림사 · 실시결과 보고서(9호) 최신 [PDF 보기] [HWP 받기]   ← 여기서 바로 끝
  ▾ 서림사 · 2026 소방계획서 [PDF 보기] [HWP 받기]
  ▾ 서림사 · 배치확인서 ⚠ 미업로드 [업로드]
  ▾ 서림사 · 소방계획서 생성 요청
```
→ **타이핑 2자 + 클릭 1번 = 다운로드/업로드/생성 완료.** 자주 하는 일은 문서 현황을 펼칠 필요조차 없다.
- **Ctrl+K 전역 팔레트**로 승격: 어느 화면에서든 Ctrl+K(모바일: 헤더 🔍)로 같은 검색 — "보고서 센터로 이동"조차 생략.
- **초성 검색**(ㅅㄹㅅ→서림사): 한글 자모 매칭 유틸 1개 — 모바일에서 타이핑 절반.

**(2) 처리 필요 목록 = 행동 내장** — 대시보드 위젯·누락 뱃지를 펼치면 각 행에 [업로드]/[바로 생성]이 있어 그 자리에서 처리. 대시보드 기준 2클릭(뱃지 → 행 버튼).

**(3) 드래그&드롭 업로드** — 문서 현황·타임라인의 업로드 행은 드롭존: 파일을 끌어다 놓으면 업로드 완료(클릭 0). 클릭 시엔 파일 선택 즉시.

**(4) 점검 완료 → 다음 행동 자동 제안** — 점검을 완료 처리하면 토스트: "별지 9호를 바로 생성할까요? [생성]" — 완료 이벤트가 다음 행동을 가져오므로 화면을 찾아다니지 않음. (선택 설정: "완료 시 9호 자동 생성" 토글 — 켜면 클릭 0, 제출·발송은 여전히 사람이)

**(5) 확인 다이얼로그 다이어트** — fail-soft 확인("빈 칸으로 생성?")은 **세션당 1회**만, 이후 같은 확인 생략(행 아래 피드백으로 대체). 파괴적 행동(삭제)만 매번 확인.

**(6) 입력 자동완성 전면화** — 모든 자유 텍스트 입력에 최근 입력·표준 문구 재사용:
- 불량 메모: 최근 사용 문구 5개 칩 + defect_catalog 표준 문구 자동완성
- 검색류: 최소 글자 2→초성 포함 1~2자
- 이미 있는 것 유지: 수신기위치·구조·지붕 datalist, 프리셋, 대장 자동 채움 (4-0-11)

**클릭 수 목표표** (대표 동선 — 구현 후 E2E로 클릭 수 검증):
| 동선 | 현재 | 목표 |
|---|---|---|
| 특정 고객 9호 PDF 확인 | 4~5클릭 (센터→카드→검색→행→받기) | **타이핑+1클릭** (행동 자동완성) |
| 배치확인서 업로드 | 3~4클릭+이동 | **드롭 0클릭 / 2클릭** |
| 점검 완료 후 9호 생성 | 3클릭+화면 이동 | **1클릭** (완료 토스트) 또는 0 (자동 생성 토글) |
| 아침 감시 | 3동작 30초 | 유지 (2클릭: 위젯→행 처리) |
| 불량 1건 입력 | 4동작 | **1탭** (단골 칩 ⑫-a) |

### ⓪ 보고서 센터 첫 화면 구조 (IA — 2026-07-26 재검토 신설)

**원칙: 검색이 곧 시작, 첫 화면은 "오늘 할 일".** 기능이 늘어나도 처음 연 사람이 헤매지 않도록 화면 골격을 먼저 고정한다:

```
보고서 센터
[ 🔍 고객명을 검색하세요 — 문서 확인·생성이 여기서 시작됩니다 ]   ← 첫 시선 = 검색창 하나
  최근 조회: [서림사] [별그리다] [행복마을]

⚠ 처리 필요 2건: 제출 기한 임박 1 · 배치확인서 누락 1   ← 접힌 뱃지(⑦), 클릭 펼침

[소방계획서] [실시결과 보고서(9호)] [이행계획·완료(10·11호)] [배치확인서]   ← 서식 카드(기존)

(기본 표시) 최근 문서 ─ 오늘 / 어제 / 이번 주 …(④)
```

- **검색하면** → 자동완성에 고객뿐 아니라 **문서·행동 후보**가 바로 뜸(4-0-13-(1) — 최신 PDF 보기·HWP 받기·업로드·생성을 드롭다운에서 즉시 실행). 고객명을 선택하면 문서 현황(①)이 아래 펼쳐짐.
- **서식 카드를 누르면** → 문서 종류 중심 흐름(소방계획서 다중 생성, 9호/10·11호 점검 건 목록 ②③) — "고객은 모르고 문서부터 찾는" 경우용.
- **아무것도 안 누르면** → 처리 필요 뱃지 + 최근 문서 — 들어오기만 해도 오늘 할 일이 보임.
- 검색창은 화면에 **하나만** 존재(②③ 카드 안의 별도 고객 검색 제거 — 카드 목록은 최근 점검 건 브라우징 + 상태 필터로 단순화). 같은 목적의 입력창이 두 개 있으면 직감이 깨진다.

### ⑤ 전/후 사진 갤러리 (2026-07-26 재설계 — 직감 동선 확정)

**원칙: 새 페이지를 만들지 않는다.** 사진은 항상 "그 점검의 불량"에 속한 데이터이므로, 삽입의 집은 지금처럼 불량내역이고 갤러리는 어디서든 열리는 **모달**이다. 사용자 멘털 모델을 한 문장으로:

> **"사진은 불량내역에서 올리고, 갤러리에서 모아 보고, 비어 있으면 그 자리에서 채운다."**

**(1) 삽입(업로드) — 기존 위치 유지 + 현장 촬영 강화**
- 원천 입력처는 현행 그대로: 점검 상세 > 불량내역 > 각 불량 행의 [전]/[후] 슬롯 (photo_url/after_photo_url — 이미 구현).
- 보강: 모바일에서 슬롯 탭 시 **카메라 바로 실행**(input capture) — 현장에서 "찍기→붙기"가 한 동작.
- 슬롯 라벨을 "사진 추가" → **"전(불량) 사진" / "후(조치) 사진"**으로 — 어느 쪽을 찍는 건지 헷갈림 제거.

**(2) 모아보기 — 갤러리 모달 (진입점 3곳, 같은 모달 재사용)**
- ⓐ 점검 상세 불량내역 헤더: **[전/후 사진 모아보기 (3/5쌍)]** — 실무자가 가장 자주 여는 곳, 쌍 수가 버튼에 미리 보임.
- ⓑ 타임라인 ⑤의 "전후 사진 n/n쌍" 텍스트 자체를 클릭 가능하게.
- ⓒ 보고서 센터 고객 문서 현황(①)의 [사진 보기].
```
┌ 전/후 사진 (3/5쌍) ─────────────────────── ✕ ┐
│ 감지기 오작동 [조치완료]     유도등 소등 [계획]  │
│ ┌전 사진┐ ┌후 사진┐        ┌전 사진┐ ┌╌╌╌╌╌┐ │
│ │  📷  │ │  📷  │        │  📷  │ ┆+ 후  ┆ │ ← 빈 슬롯 = 앰버 점선,
│ └──────┘ └──────┘        └──────┘ ┆ 사진 ┆ │    탭하면 그 자리에서 업로드
│                                    └╌╌╌╌╌┘ │
└──────────────────────────────────────────────┘
```
- **빈 슬롯을 갤러리 안에서 바로 업로드** — "보다가 비면 그 자리에서 채운다"가 핵심 직감. (불량내역으로 돌아갈 필요 없음, 같은 업로드 액션 재사용)
- 페어 카드에 불량명 + 조치 상태 뱃지(계획/완료) — 사진만 보고 진행 상황 파악.
- 클릭 확대, ←→/스와이프 넘기기, ESC 닫기. URL 해시(#photos) 딥링크로 공유 가능.

**(3) 쓰임 — 자동 (사용자 행동 불필요)**

- 타임라인 ⑤·문서 현황의 "n/n쌍" 카운트가 같은 데이터를 읽으므로 어디서 채워도 모두 갱신.

### ⑫ 불량 입력 편의 개선 (2026-07-26 사용자 요청 — "불량 입력을 조금 더 편리하게")

현재 동선: 검색(2자)→후보 선택→메모 타이핑→[불량 저장] = 건당 4동작, 사진은 불량내역에서 별도. 이를 아래 4가지로 줄인다:

**⑫-a. 단골 불량 원터치 칩 (핵심 — 검색 생략)**

  예) `[소화기 압력 미달] [유도등 점등 불량] [감지기 오작동] [비상구 적치물] …`

**⑫-b. 연속 입력 모드 (여러 건 쭉쭉)**
- 저장 후 폼을 닫지 않고 검색창 초기화+포커스 유지 — 다음 건 바로 입력.

**⑫-c. 사진 즉시 첨부 (찍고 저장 한 흐름)**

**⑫-d. 점검표 시트 X 인라인 등록**

**A. [통합] 보고서 센터에 "제출 현황" 판 신설 → 구 현황판 2개 흡수·폐지**

- 데이터 소스 = **타임라인 필드 단일**(report9/11_submitted_at·report_deliveries·defects·cert 슬롯). 수기 입력 없음 — 타임라인에서 일하면 현황판은 저절로 채워진다.
- `/inspection-reports/status`·`/action-plans/status` → 보고서 센터 제출 현황으로 **리다이렉트**.

**B. [통합] 이행계획서 등록 화면 폐지 → 점검 타임라인이 대체**
- 신규 건은 이미 타임라인 ⑤⑥ + 불량내역 조치 필드(§9-7a)로 처리 가능 — `/action-plans`는 안내 후 점검 목록으로 리다이렉트.
- 기존 수기 데이터(action_plans 3종·inspection_report_status): **1회 마이그레이션**으로 타임라인 필드에 이관(제출일·발송일 등 매핑 가능분), 이관 불가분은 읽기 전용 보존. ⚠ 데이터량·충돌은 착수 시 조사 후 사용자 확인.

**C. [유지] 점검현황 모니터링 · 점검대장**
- 문서와 중복 없음(일정·대장 도메인). 수정은 소폭: 용어 통일(⑪) + 행에서 점검 상세·보고서 센터 문서 현황으로 딥링크 + D-day 색 규약 정렬.

**D. 사이드바 정리 — 문서 메뉴 5개 → 2개**
```
(전) 점검현황 모니터링 / 점검대장 / 보고서 제출현황 / 이행계획서 등록 / 이행계획 제출현황 / 보고서
(후) 점검현황 모니터링 / 점검대장 / 보고서   ← 문서는 "보고서" 하나 (직감 지도 4-0-8과 일치)
```

### 세션 반영
- **S4 (신설)**: A(제출 현황 판+리다이렉트) → B(등록 화면 폐지+데이터 이관 조사→사용자 확인→이관) → D(사이드바 정리, 7-B 최종 메뉴안 포함) → C(소폭 수정) → 안전관리 대장→정산현황 탭 흡수(7-B). 예상 1일 추가.
