추가 개발건 테스트 — 스테이징 DB 분리를 권장합니다

지금 구조의 문제: 로컬 개발과 운영 서버가 같은 Supabase DB를 공유합니다. 지금까진 전부 테스트 데이터라 괜찮았지만, 실데이터가 들어가는 순간부터 로컬에서 기능을 돌려보는 것 자체가 운영 DB를 오염시킵니다.

권장 구성 (무료):

1. Supabase 무료 프로젝트를 하나 더 만들어 스테이징으로 사용 (무료 티어로 2개까지 가능)
2. 마이그레이션 001~047을 스테이징에 일괄 적용 (제가 _apply-sql.mjs로 자동 적용 가능)
3. 기존 mock 데이터를 스테이징에 재시딩 — 어제 만든 cleanup 백업 JSON(556건)을 복원하거나 기존 시드 스크립트를 재실행하면 테스트 환경이 그대로 재현됩니다
4. erp/.env.local을 스테이징 키로 교체 → 로컬 dev = 항상 스테이징, 서버 = 운영 으로 완전 분리
5. victory_test 검증 스크립트들이 전부 .env.local을 읽으므로 수정 없이 스테이징 대상으로 동작합니다

이후 개발 흐름: 로컬(스테이징 DB)에서 개발·테스트 → 마이그레이션도 스테이징 먼저 검증 → push → VPS 배포 → 운영 스모크. 운영 DB에는 검증된 것만 닿습니다.

말씀만 주시면 스테이징 프로젝트 생성부터 마이그레이션 적용·시드 복원·env 전환까지 한 번에 세팅해 드리겠습니다.
