### 2-2. 검출력 테스트 시나리오 (INV-DET) — "불변식이 진짜 잡아내는가"

각 불변식에 위반 데이터를 **의도적으로 주입 → 스크립트가 검출 → 정리** 하는 3단계. 임시(TEST-) 고객으로만 수행.

| # | 대상 | 주입 방법 (service role) | 기대 검출 | 상태 |
|---|---|---|---|---|
| DET-1 | INV-P1 | 임시 고객에 지난 달 정기 planned 항목 직접 insert | INV-P1 1건 | pending |
| DET-2 | INV-P2 | planned_date < 기준일 항목 insert | INV-P2 1건 | pending |
| DET-3 | INV-P3 | notes에 ⟦자동취소:planned⟧ 넣고 status=planned로 insert | INV-P3 1건 | pending |
| DET-4 | INV-P4·P5 | 비활성 임시 직원을 담당으로 하는 planned 항목+고객 구성 | P4·P5 각 1건 | **passed** — 조중민 실사례에서 17+1건 검출 실증 |
| DET-5 | INV-P6 | 진행중 점검의 항목 담당만 다른 직원으로 update | INV-P6 1건 | **passed** — 탑텐 실사례 검출 실증 |
| DET-6 | INV-1 | 임시 고객 비활성 처리 후 항목 1건을 planned로 되돌림 | INV-1 1건 | pending |
| DET-7 | INV-3 | completed 항목의 inspection_id를 null로 update | INV-3 1건 | pending |
| DET-8 | INV-4 | confirmed 항목의 step1_date를 null로 update | INV-4 1건 | pending |
| DET-9 | INV-5 | cancelled 항목에 step1~6_date 값 주입 | INV-5 1건 | pending |

> DET 자동화 시 주의: 주입→검사→정리를 한 스크립트의 try/finally로 묶고, 검사 단계는 두 불변식
> 스크립트를 자식 프로세스로 실행해 exit code 1(위반 검출)을 **성공 조건**으로 판정한다.
