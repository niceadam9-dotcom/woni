# 점검계획일 기준 표시·동기화 개선 테스트 시나리오 (TS-ANCHOR)

작성: 2026-07-14. 같은 날 구현한 "점검계획일 기준 통일" 개선(변경전파맵 1-1·1-5·1-5b·1-11)의 검증 시나리오.
배경: 운영/스테이징에서 "화면마다 사용승인일·점검계획일이 다르게 보이고 고객관리와 점검확정 일정이 안 맞는다" 문의
→ 원인 5건 수정 (①계획 화면의 사용승인일 라벨에 기준일 치환 표시 ②점검유형 변경 시 계획 항목 미동기화
③updateCustomerAction 부분 호출 시 미전송 필드 null 덮어쓰기 ④점검 등록 트리거의 점검계획일 누락 ⑤점검계획일 안내 부족).

**공통 원칙**
- 테스트 데이터는 `TEST-` 접두 임시 고객으로 만들고 종료 시 전부 정리. **스테이징에서 먼저 수행** 후 운영 배포
- 화면 확인과 DB 교차검증을 병행. 자주 쓰는 확인 쿼리:
  - 고객: `select customer_name, inspection_type, contract_date, use_approval_date, plan_anchor_date, address from customers where id = :id`
  - 계획 항목: `select p.year, p.month, i.sequence_num, i.plan_type, i.inspection_type, i.inspection_sub_type, i.planned_date, i.scheduled_date, i.status, i.inspection_id from inspection_plan_items i join inspection_plans p on p.id = i.plan_id where i.customer_id = :id order by p.year, p.month, i.sequence_num`
- "반영 화면" 전부에서 확인: 고객관리 목록·상세, 점검확정(목록·달력), 점검달력, 점검대장

## A. 표시 일관성 (원인 ①)

### TS-ANCHOR-1. 점검확정 "점검계획일" 컬럼 = 고객관리와 동일한 원본 값
- **사전조건**: 임시 고객 TEST-A — 종합, 사용승인일 `2016-07-29`, 점검계획일 `2026-03-05` (서로 다른 값·다른 월)
- **단계**: ① 점검확정 목록에서 TEST-A 행의 컬럼 헤더가 "점검계획일"(사용승인일 아님)인지
  ② 그 값이 `2026-03-05` — 고객관리 목록·상세의 점검계획일과 동일한지
  ③ 고객관리의 사용승인일은 `2016-07-29` 그대로(계획 화면 값과 섞이지 않음)
  ④ 점검대장에서도 점검계획일/사용승인일 두 컬럼이 각각 원본인지
- **프로브(회귀 핵심)**: TEST-A에 점검 1건 등록(시작일 임의) → 점검확정의 점검계획일 표시가 **바뀌지 않아야** 함
  (구버전은 기준일 치환 때문에 점검 등록 시 "사용승인일"이 바뀌어 보였음 — 이번 문의의 재현 케이스)

### TS-ANCHOR-2. 미입력 고객 표시 + 인라인 입력 유도
- **사전조건**: 점검계획일 NULL 임시 고객 TEST-B (SQL로 필수화 이전 상태 재현: `update customers set plan_anchor_date = null where id = :id`)
- **단계**: ① 고객관리 목록·점검확정 목록에서 빨간 **"미입력"** 표시 확인 (관리 권한 계정)
  ② 점검확정의 미입력 클릭 → 인라인 날짜 입력 → 저장 → 값 반영 + planned 항목 예정일이 새 기준일의 '일'로 재계산
  ③ 일반 직원(비관리) 계정으로 접속 → 인라인 편집 없이 "미입력" 텍스트만 표시
  ④ **슬라이드 패널**: 항목 행 클릭 → 패널 상단 "점검계획일 (계획 기산일 · 고객관리와 동기화)" 필드 표시.
  관리 권한이면 여기서도 인라인 편집 가능 — 수정 시 목록·고객관리와 동일 값, 확정 보유 시 B안 팝업 동일 작동
- **프로브**: 운영 실데이터 3곳(힘찬해가·탑텐·샤브올데이) — 배포 후 실제 입력 처리
- **자동화**: 브라우저 패턴으로 가능

### TS-ANCHOR-3. 라벨·자동 계산 전수 확인 (모달 4종)
- **단계**: ① 일정 자동 제안 모달 — 부제 "점검계획일(기준일) 기준", 그룹 제목 "점검계획월 N월",
  제안 사유에 기준 출처(점검계획일/사용승인일/점검시작일) 표기, 추가 시 예정일 = 기준일 '일'의 해당월 다음 영업일
  ② 미점검 초과 해결 모달 — "점검계획일 기준" 문구 + 항목별 "점검계획일 YYYY.MM.DD"
  ③ 항목 추가 모달 — 고객 선택 시 "자동 — 점검계획일 다음 영업일" + 날짜 자동 채움(기준일 기반)
  ④ 계획 자동생성 위저드 — 날짜 자동 배분이 기준일 '일' 기준으로 계산되는지
- **프로브**: 점검계획일 없는 TEST-B(폴백 기준일)도 제안·자동 계산이 동작해야 함 (기준일 폴백 유지 확인)

## B. 점검계획일 필수화 (원인 ⑤)

### TS-ANCHOR-4. 비우기 차단 (3경로)
- **단계**: ① 고객 상세 수정 폼 — 점검계획일에 빨간 * 필수 표시 + "등록일이 아닌 연간 점검의 기산일…" 힌트 확인.
  값 지우고 저장 → "점검계획일은 필수입니다" 에러, 저장 안 됨
  ② 고객관리 목록 인라인 편집 — 값 지우고 저장 → "점검계획일은 필수값입니다 — 비울 수 없습니다" 알림, 원값 유지
  ③ 신규 등록 폼 — 점검계획일 없이 제출 → 차단 + 힌트 문구 확인
- **프로브(서버 방어)**: 서버 액션 직접 호출 수준에서도 거부되는지 (updateCustomerAction에 `plan_anchor_date: ''` 전달 시 에러 반환)

## C. 기준일 변경 B안 팝업 (변경전파맵 1-5)

### TS-ANCHOR-5. 확정 일정 보유 + "확정해지 후 전체 재계산"
- **사전조건**: 임시 고객 TEST-C — 점검계획일 3월 10일, planned 3건 + confirmed 1건(점검 미시작, scheduled_date 있음)
- **단계**: ① 고객관리(인라인 또는 수정 폼)에서 점검계획일 3월 10일 → 9월 22일 변경
  ② 저장 전 팝업 표시 — 확정 항목 1건의 연·월·확정일 목록 정확한지, **이 시점 DB는 아직 원값**인지(취소 대비)
  ③ "확정해지 후 전체 재계산" 선택 → 고객 점검계획일 9월 22일 반영
  ④ confirmed였던 항목: status=planned, scheduled_date=null, step1~6_date=null(DB)
  ⑤ 모든 planned 예정일이 22일 기준(영업일 회피 포함)으로 재계산 ⑥ 점검달력 위치 이동 확인
- **자동화**: 브라우저 패턴 필요(서버 액션 세션)

### TS-ANCHOR-6. "확정 유지" / "취소" 분기
- **사전조건**: TS-ANCHOR-5와 동일 구성 (TEST-D)
- **단계**: ① 동일 변경 → 팝업에서 **"확정 유지 — 미확정 일정만 재계산"** 선택
  → 점검계획일 변경됨 + planned만 재계산 + confirmed의 scheduled_date·단계 일정 **불변**
  ② 다시 변경 시도 → 팝업에서 **"취소"** 선택 → 고객 점검계획일·계획 항목 전부 원값 그대로(아무것도 저장 안 됨)
- **프로브**: 사용승인일 변경도 같은 팝업 플로우를 타는지 (기준일 폴백 대상 고객: 점검계획일 NULL 상태에서 사용승인일 변경)

### TS-ANCHOR-7. 점검 시작된 확정 항목은 해지 대상 제외
- **사전조건**: TEST-E — confirmed 2건: 하나는 점검 시작됨(inspection_id 연결), 하나는 미시작
- **단계**: ① 점검계획일 변경 → 팝업 목록에 **미시작 1건만** 표시
  ② "확정해지 후 전체 재계산" → 미시작 건만 planned 복귀, 시작된 건은 상태·일정 불변
  ③ 시작된 점검의 6단계 마감일도 불변인지 점검 상세에서 확인

## D. 점검유형 변경 동기화 (원인 ②, 변경전파맵 1-5b)

### TS-ANCHOR-8. 종합→작동 (환희텔 사례 재현)
- **사전조건**: TEST-F — 종합, 연간 계획 자동 생성됨(1차 special_종합 + 2차 special_종합(미래월) + monthly), 그중 monthly 1건은 confirmed
- **단계**: ① 고객 상세의 점검유형 변경 모달에서 종합→작동
  ② planned 항목: inspection_type=작동, inspection_sub_type=작동, special_종합→**special_작동** (DB)
  ③ planned 상태의 **2차(seq2) 특별점검 삭제** ④ confirmed 항목은 유형·일정 불변
  ⑤ 점검확정 화면 유형 배지가 작동으로 표시 (구버전 버그: special_종합 잔존)
- **프로브(양 경로)**: 같은 시나리오를 고객관리 목록 **인라인 편집**으로도 반복 — 결과 동일해야 함.
  인라인 변경 시 customers.inspection_category/sub_type도 갱신되는지(이번에 함께 수정된 버그)

### TS-ANCHOR-9. 작동→종합 (2차 보충 생성)
- **사전조건**: TEST-G — 작동, 점검계획일의 월이 현재월 이전~+5개월 내(2차가 올해 미래월에 생기도록)
- **단계**: ① 작동→종합 변경 ② planned: special_작동→special_종합, sub_type=종합
  ③ **2차 특별점검(+6개월) 신규 생성** — 미래월만, 기준일 이전/지난 달은 생성 안 됨
  ④ 한 번 더 작동→종합 반복(중간에 작동으로 되돌린 뒤) → 항목 중복 없음(멱등, UNIQUE 스킵)
- **프로브**: 2차 예정 월이 이미 지난 과거(기준일 월+6 < 현재월)면 생성되지 않는 것이 정상

### TS-ANCHOR-10. 일반관리 전환 왕복
- **사전조건**: TEST-H — 종합(연간 계획 있음, confirmed 1건 포함)
- **단계**: ① 종합→일반관리 → 소방안전관리 **planned 삭제**, confirmed·완료 불변, 고객 category=일반관리
  ② 일반관리→종합 → 올해 연간 항목 재생성(특별+정기, 지난 달 정기 제외)
  ③ 일반관리 고객의 수동 등록 점검 이력이 있다면 불변인지

## E. 부분 업데이트 안전성 (원인 ③ — 데이터 소실 회귀)

### TS-ANCHOR-11. 점검유형만 변경해도 다른 필드 보존
- **사전조건**: TEST-I — 계약일·사용승인일·점검계획일·주소·우편번호·비고·관할서 전부 값 있는 고객
- **단계**: ① 고객 상세 점검유형 변경 모달로 종합→작동 저장
  ② DB에서 **contract_date, use_approval_date, plan_anchor_date, address, zipcode, region_*, notes, fire_station 전부 원값 보존** 확인
  ③ 활동 로그(변경 이력)에 "점검유형 종합→작동" **1건만** 기록 — 날짜 필드의 허위 변경 이력 없음
  ④ 고객관리 목록 인라인으로도 반복 확인
- **프로브(비우기 동작 유지)**: 수정 폼에서 사용승인일을 **의도적으로 지우고** 저장 → null 저장(비우기는 폼 경유만 허용).
  계약일·비고도 동일
- **자동화 권장**: 이 케이스는 회귀 위험이 커서 스크립트화 가치 높음

## F. 점검 등록 트리거 (원인 ④, 변경전파맵 1-11)

### TS-ANCHOR-12. 점검계획일 있는 고객 — 재생성 없음
- **사전조건**: TEST-J — 종합, **사용승인일 없음 + 점검계획일 있음**(등록 시 연간 계획 생성됨). 생성 직후 항목 수 기록
- **단계**: ① 점검업무에서 TEST-J 점검 직접 등록(시작일 = 점검계획일과 **다른 월**)
  ② 계획 항목 수 **불변** — 특히 2차 특별점검이 다른 달에 중복 생성되지 않음 (구버전 버그 재현 조건)
  ③ 기존 항목의 예정일·plan_type 불변
- **DB 교차검증**: `select count(*) ... where customer_id=:id and sequence_num=2` 가 등록 전후 동일

### TS-ANCHOR-13. 둘 다 없는 고객 — 점검시작일 기준 생성 (기존 동작 유지)
- **사전조건**: TEST-K — 종합, 사용승인일·점검계획일 모두 NULL(SQL로 구성), 계획 항목 0건
- **단계**: ① 점검 직접 등록(시작일 임의) ② 점검시작일을 기준으로 연간 항목 생성(1차 특별=시작월, 지난 달 정기 제외)
  ③ 같은 고객에 점검 재등록 시도(다른 차수) → 항목 중복 없음(멱등)
