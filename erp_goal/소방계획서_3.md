## 1. 탭 전체 레이아웃

```
고객관리 > 소방계획서 탭
┌─────────────────────────────────────────────────────────┐
│ [상단 고정 바]                                            │
│  준비율 게이지 14/17 ▓▓▓▓░  · 마지막 저장 07-21 14:30 김직원 │

```

- **아코디언 방식**(기존 "계획서 정보" 패널 패턴 재사용). 섹션 제목 = 한글 서식 번호·명칭 그대로 → 사용자가 한글 문서와 화면을 1:1 대응 가능.
- 섹션마다 우측에 완성도 뱃지(✓ / n/m / 미입력 / 자동). 준비율 게이지는 섹션 뱃지의 합산으로 재정의(기존 fire-plan-readiness 확장).
- 각 섹션은 **개별 저장**(섹션 하단 저장 버튼 + 변경 감지 시 강조). 저장 대상은 섹션별로 기존 정규 테이블 또는 신규 `fire_plan_forms` JSONB(아래 §3).
- 기존 "표준양식 생성" 모달은 폐지. 모달의 입력 항목들은 아래 해당 섹션으로 흡수.

## 2. 섹션별 상세 구성

표기: `[기존]` = 이미 있는 데이터/화면 이관, `[신규]` = 새로 개발, `저장:` = 저장 위치

### 2-1. 서식 1.1 건축물 일반현황 `[기존 이관]`
- 필드: 명칭(고객명 자동) · 도로명주소 · 등급(특급/1급/2급/3급 세그먼트) · 용도 · 사용승인일 · 연면적/건축면적(㎡) · 층수(지하/지상) · **높이(m)** · 구조 · 지붕 · **수신기 위치** · 대표자(성명·연락처) · 소방안전관리자(성명·연락처·선임일)
- 저장: `customers`(급수·선임일 등) + `buildings`(면적·층수·높이·구조·지붕·수신기) + `customer_contacts`(대표)
- 비고: 요구 3·4·5의 버그(웹 하드코딩)를 이 섹션 구현 시 함께 해소. 건물·시설 탭과 중복 편집 가능하나 저장소가 같아 충돌 없음

### 2-2. 서식 1.2 건축물 세부현황 `[모달에서 이관]`
- **1.2.1 구역별 세부현황**: 행 목록 — 구역 / 명칭 / 면적 / 근무인원(평일·휴일) / 관리업체 / 연락처 (+층 자동 생성 버튼: fire_facility_floors 기반)
- **1.2.2 화재취약장소**: 행 목록 — 장소 / 위치 / 위험요인(전기적·기계적·화학적·가스누출·부주의·자연재해 다중 체크) — 프리셋(보일러실·주방·전기실) 기본 제공
- 저장: `fire_plan_forms.sections.zones` / `.hazards` (JSONB)

### 2-3. 서식 1.3 소방서 일반현황 `[모달에서 이관]`
- 필드: 관할 소방서(자동완성 — 요구 7·8) · 거리(km) · 도착 예상(분)
- 저장: `customers.fire_station` + `fire_plan_forms.sections.station`(거리·시간)

### 2-4. 서식 1.4 소방시설 현황 `[기존 이동 + 재구성]` (요구 10·23)
- 건물·시설 탭의 소방시설현황 패널을 **이 섹션으로 이동**하고, 항목 구성을 **서식 1.4의 분류·명칭 그대로** 재정렬(소화설비/경보설비/피난구조설비/소화용수설비/소화활동설비), 체크박스 클릭 방식
- 층별 수량(fire_facility_floors)은 하위 접기로 유지
- 저장: `fire_facilities` / `fire_facility_floors` (기존 테이블 그대로)
- 비고: 서식 1.4 항목명과 현행 facility_code 매핑표를 먼저 확정(불일치 항목 정리 — 요구 10의 "매치 안 됨" 해소)

### 2-5. 서식 1.5 피난·방화·제연·방염 `[신규]` (요구 11·12)
- **1.5.1 일반현황** (양식 추출 기준):
  - 피난시설 > 계단: 직통계단 / 피난계단 / 특별피난계단 / 옥외계단 (체크 + 개수)
  - 피난시설 > 기타: 대피공간 / 경량칸막이 / 피난안전구역 / 옥상광장 / 기타(텍스트)
  - 피난층: 위치(층) · 출입구 개소 · 개폐방법
  - 방화구획: 해당유무 / 면적별 / 층별 (체크)
  - 방화문·제연설비·방염대상물품: 양식 원본 항목 그대로 체크·수량 (구현 시 원본 재확인)
- **1.5.2 방화·제연구획 현황도**: 구역별 [층 라벨 + 평면도 이미지 업로드 + 설명] 행 목록. 연면적 1,000㎡ 이상 시 작성 안내 문구
- 저장: `fire_plan_forms.sections.evacFire` + 도면 스토리지 `{customerId}/plan-assets/1_5_2_{stamp}.png`

### 2-6. 서식 1.6 기타시설 현황 `[신규]` (요구 13)
- 전기시설: 수전용량(kW) · 변압기용량(kVA) · 설치위치 · 수량(대) · 비상발전기(kW·설치위치·수량) · 비고
- 가스시설: 가스종류(LPG/LNG 등) · 사용위치 · 용도 · 정압기 위치 · 차단기구(유/무) · 차단기 위치 · 비고
- 위험물·기타: 해당없음 체크 포함, 양식 항목 그대로
- 저장: `fire_plan_forms.sections.etcFacility`

### 2-7. 서식 1.7 선임현황 + 자위소방대 `[기존 확장]` (요구 15)
- **1.7.1 선임현황** 행 목록: 구분(소방안전관리자/보조자) / 소속 / 선임자 성명 / 선임일자 / 실무교육 수료일자 / 담당업무
  - 1행(관리자)은 customers 데이터 자동, 추가 행(보조자 등)은 신규 입력
- **자위소방대 편성표**(제2장, 기존): 팀 / 성명 / 임무 / 연락처 — 기존 fire_brigade_members 편집 UI 이관 + **비상연락체계** 필드 추가
- 저장: `fire_brigade_members`(컬럼 확장: affiliation·edu_completed_at·assignment 등) 또는 선임현황만 `fire_plan_forms.sections.managers` — 구현 시 결정(추천: 선임현황은 JSONB, 편성표는 기존 테이블 유지)

### 2-8. 서식 1.8 업무대행 현황 `[자동]`
- 회사 정보(company_profile) + 계약일 + 등급 체크 자동 표시 — **읽기 전용 미리보기**만 제공(수정은 관리자 설정에서)

### 2-9. 서식 1.10 자체점검 `[신규 + 일부 자동]` (요구 17·19·20)
- **1.10.1 연간 점검 계획**:
  - 작동점검: 점검시기(년·월 — 점검계획일 기반 자동, 수정 가능) · 점검자(자체/외주)
  - 종합점검(종합 대상만): **최초점검 체크 + 년·월**(요구 17) / 종합점검 년·월 / 종합 2차(특급) / 점검자
  - 사용승인일 자동 표시, 결과보고·제출처(관할서) 자동
- **1.10.3 다중이용업소 관리현황**: "다중이용업소 해당" 토글 → 해당 시에만 입력 표 노출 — 사업장명 / 업종 / 위치 / 영업주 / 연락처 / 영업시간(평일·휴일 × 주간·야간) / 이용자 유형(노유자·주취자·청소년·신체부자유자) / 수용인원 / 안전점검
- **1.10.4 화재/비화재보 이력**: 행 목록 — 구분(화재/비화재보) / 발생일시 / 발생장소 / 발생원인 / 조치사항
- 저장: `fire_plan_forms.sections.inspection` / `.multiUse` / `.fireHistory`
- 비고: "소방안전관리 업무수행"(요구 18)은 1.10.2 업무수행 기록 — 수기 성격이 강해 v1 제외 검토(확인 질문 유지)

### 2-10. 서식 1.11 소방훈련·교육 `[신규]` (요구 21·22)
- **1.11.1 연간계획**: 대상자 인원(근무자·거주자·자위소방대 명수) + 구분(소방교육/소방훈련 등) × 월(1~12) 체크 그리드
- **1.11.2 세부계획**: 차수별 행 — 명칭 / 일시 / 장소 / 대상(자위소방대·근무자·거주자) / 종류(실습: 기본·부분·종합, 이론: 강의·세미나) / 훈련형태(자체·합동) / 시나리오 요약 / 교보재 / 훈련·교육·평가계획
- **1.11.3 소방훈련 시나리오**: 텍스트 영역 + **유형 프리셋 불러오기 버튼**(주택형/상가형/공장형 — 기존 프리셋과 연동, 불러온 뒤 고객별 수정 저장)
- **1.11.4 실시 결과 기록부**(별지 28호, 2년 보관): 실시일 / 구분 / 참여인원 / 내용 / 평가 행 목록 — v1은 입력·보관, 서식 출력은 후순위
- 저장: `fire_plan_forms.sections.training`

### 2-11. 서식 3.5 피난약자 현황·피난계획 `[신규]` (요구 16)
- 근무·거주자 현황: 노인 / 어린이 / 영유아 / 임산부 / 장애인 / 기타 — 각 체크 + 명수
- 시설이용자: 동일 유형 체크

- 저장: `fire_plan_forms.sections.vulnerable`

## 3. 데이터 모델

### 3-1. 신규 테이블 `fire_plan_forms` (마이그레이션 096 예정)

```sql
CREATE TABLE fire_plan_forms (
  customer_id UUID PRIMARY KEY REFERENCES customers(id) ON DELETE CASCADE,
  sections    JSONB NOT NULL DEFAULT '{}',   -- 섹션 키별 저장 (아래)
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_by  UUID REFERENCES profiles(id)
);
```

섹션 키: `zones` `hazards` `station` `evacFire` `etcFacility` `managers` `inspection` `multiUse` `fireHistory` `training` `vulnerable` `evacPlan`(집결지·유도방법)

- **원칙**: 이미 정규 테이블이 있는 데이터(고객·건물·시설·자위소방대)는 JSONB에 **중복 저장하지 않는다**. JSONB는 서식 전용 항목만.
- 저장 API: 섹션 단위 upsert 액션 1개(`saveFirePlanSectionAction(customerId, key, value)`) — zod 스키마로 섹션별 검증.

### 3-2. 생성 연결 (어댑터)

- `assembleFirePlanGenData(customerId)`: 정규 테이블 + fire_plan_forms → `FirePlanGenData`(확장) 조립 — 웹 PDF·HWP 워커·데이터시트가 공유
- 생성 시 `.form.json` 스냅샷은 현행대로 보존(이력·재생성용)
- 최초 진입 시 fire_plan_forms가 비어 있고 최신 `.form.json`이 있으면 **1회 임포트** 제안(기존 고객 재입력 방지)
- 신규 섹션의 HWP 병합은 placeholder 확장(ANCHOR_KEYS 추가 + 심기 재실행) — 섹션별로 순차 적용, 미적용 섹션은 웹 PDF에만 먼저 반영

## 4. 준비율·경고 연동

- 탭 뱃지(`14/17`)를 섹션 완성도 합산으로 재정의: 필수 섹션(1.1~1.4, 1.7, 1.10)은 미입력 시 경고색, 선택 섹션(1.5·1.6·다중이용업소 등)은 "해당 없음" 체크 시 완료로 간주
- 생성 버튼 옆에 누락 요약("1.5 미입력 · 1.10 최초점검 미기재") — 누락 있어도 생성 가능(빈 칸 출력) 원칙 유지
