# 소방계획서 HWP 자동 생성 시스템 정리

> 최종 갱신: 2026-07-21 (DB 큐 전환 · 2단계 등록 · HTML 미리보기 반영)
> 관련 설계: `소방계획서-필드확장-설계.md`, `Victory_md/01_flow/doc02_승진소방_실무연결_상세설계.md` §8

## 1. 개요

고객·건물·시설 데이터를 표준양식(25년 이후 소방계획서 양식)에 자동 병합해
**HWP(한글 원본) + 웹 미리보기(HTML) + 인쇄용 PDF**를 생성하고 고객 상세의 소방계획서 보관함에 등록하는 시스템.

- 한글 SDK가 Windows 전용이라 생성은 **사무실/개발 PC의 상주 워커**가 담당하고, ERP 웹은 요청·상태 표시만 한다.
- 사용자는 요청 후 **몇 초 안에 HWP·미리보기**를 받으며, PDF는 뒤따라 자동으로 채워진다(2단계 등록).

## 2. 구성 요소

| 구성 | 위치 | 역할 |
|---|---|---|
| 생성 요청 페이지 | `/fire-plans/generate` | 고객 검색(다중) → 연도·프리셋 선택 → 요청. 처리 중/대기/결과 5초 폴링 표시 |
| 서버 액션 | `erp/src/app/(dashboard)/fire-plans/generate/actions.ts` | 큐 등록(중복 차단)·상태 조회·프리셋 관리 |
| DB 큐 | `fire_plan_gen_jobs` (마이그레이션 094) | pending → processing → done/failed. 같은 고객·연도 활성 요청 1건 유니크 |
| 워커 하트비트 | `fire_plan_worker_status` (단일 행 id=1) | 90초 내 갱신 = 온라인. processing_note에 현재 작업 표시 |
| 생성 워커 | `erp/scripts/fireplan-worker.py` | 10초 폴링 → 선점(claim) → 생성·업로드·등록. Windows 전용 |
| 병합 엔진 | `erp/scripts/make-fireplan.py` | 한글 SDK 파이프라인 + CLI. 워커가 함수 import |
| PDF 변환(기본) | 로컬 LibreOffice (`soffice --headless`) | ODT → PDF, 건당 2~8초. 26.2.4 winget 설치됨 |
| PDF 변환(폴백) | `/api/cron/convert-fireplan-pdf` | 워커 실패·중단 시 VPS Gotenberg가 스토리지 ODT로 변환. 10분 주기 + 워커 즉시 트리거 |
| 보관함 | `fire_plans` 테이블 + `fire-plans` 버킷 | 고객·연도·개정판별 HWP/HTML/PDF 보관, 인쇄·다운로드·제출추적 |

## 3. 생성 흐름 (2단계 등록)

```
[요청]  ERP → fire_plan_gen_jobs INSERT (pending)
          · 같은 고객·연도의 대기/처리 중 요청은 유니크 인덱스로 차단
          · 워커 오프라인이면 요청 화면에서 confirm 경고 1단계 추가

[1단계] 워커 (몇 초)
          고객·건물·시설·관계인·자위소방대 조회
          → 양식-placeholder.hwpx {{key}} 치환 병합 (+프리셋, +표지 사진)
          → SDK SaveAs: HWP · ODT · HTML (HTML은 PIC*.png 이미지 base64 인라인 처리)
          → 스토리지 업로드 + fire_plans 등록 (pdf_status='converting')
          → 작업 done — 사용자는 이 시점부터 HWP·미리보기 사용 가능

[2단계] PDF 첨부 (뒤따라 2~8초)
          로컬 LibreOffice: ODT → PDF → 업로드 → pdf_status='ready', ODT 삭제
          실패/미설치 시: 크론이 스토리지 ODT를 Gotenberg(도커 내부망)로 변환해 채움
```

- 스토리지 경로 규약: `{customerId}/{year}/generated_hwp_{stamp}.{hwp|html|odt|pdf}`
- 개정판: 같은 고객·연도 기존 건수 +1

## 4. 자동 병합 데이터

우선순위: **고객 데이터 > 프리셋 > 양식 기본값**

- 서식 1.1: 명칭·주소·사용승인일·용도·연면적·층수·대표자·관리자·연락처·수신기위치·구조·지붕·높이
- 서식 1.2.1: 구역별 행(층별 또는 전층 폴백), 1.4: 시설현황 체크(□→■), 1.7: 선임 성명·일자
- 1.10: 자체점검 시기(종합 = 기준월, 작동 = +6개월), 급수·화재보험·운영시간·인원
- 자위소방대: 대장/부대장 행 (`fire_brigade_members`)
- 표지 사진: `{customerId}/gen-assets/` 최신 이미지 자동 삽입
- 공통 수기 프리셋(7차): `_presets/{house|retail|factory}.json` — 주택형/상가형/공장형 문구 치환, ERP 프리셋 관리에서 편집
- 누락 필드: 생성 후 라벨 목록으로 기록(`missing`) → 생성 페이지·보관함에서 "고객 정보 입력" 링크 안내. 누락이 있어도 생성 가능(빈 칸 출력)

## 5. 보관함 동작 (095 이후)

| pdf_status | 화면 표시 | 가능한 동작 |
|---|---|---|
| converting | "PDF 변환 중" 스피너 | HWP 다운로드, 웹 미리보기 |
| ready | PDF | 인쇄(자동 인쇄 대화상자)·PDF/HWP 다운로드·미리보기·연차발행 |
| failed | "PDF 실패" | HWP·미리보기 사용 가능, pdf_error에 사유 기록 |

- **웹 미리보기(HTML)**: 레이아웃 참고용 — 관공서 제출·인쇄는 반드시 PDF 사용
- 연차발행은 pdf_status='ready'일 때만 허용

## 6. 운영 가이드

- **워커 실행**: 개발 PC에서 `python scripts/fireplan-worker.py` (Ctrl+C 종료). 현재 스테이징 DB 대상(`load_env(prod=False)`)
- **워커 오프라인 시**: 요청은 접수되어 큐에 대기, 워커가 켜지면 순서대로 처리. 화면에 오프라인 뱃지 + 요청 시 경고
- **워커 중단 복구**: 재시작 시 processing 잔여 작업을 pending으로 복귀(시도 3회 초과분은 failed)
- **PDF만 실패한 경우**: 크론이 자동 재시도(ODT 보존, converting 유지). `pdf_error` 확인
- **트라이얼 라이선스**: 시작 직후 "SDK 초기화 실패"로 죽으면 트라이얼 만료(~2026-08-14 추정) — 정식 라이선스 필요
- **스모크 테스트**: 워커 가동 상태에서 `node scripts/_smoke-worker-queue.mjs` (등록→처리→PDF ready→정리 자동 검증)
