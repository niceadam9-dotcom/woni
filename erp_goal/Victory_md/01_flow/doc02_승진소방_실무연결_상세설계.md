[ERP 웹 화면]                                    [보고서 생성 파이프라인]

                                              │ 1. 데이터 수집 (고객·건물·관계인·점검인력·
                                              │    소방시설현황·점검표결과·불량)
                                              │ 2. exceljs로 갑지 템플릿(.xlsx) 셀 주입
                                              │ 3. 변환 컨테이너(LibreOffice)로 PDF 변환
                                              │ 4. Supabase Storage 저장 + 이력 기록
                                              ▼
                            브라우저: PDF 새 탭 오픈 → 자동 window.print() 호출

## 1. DB 설계 — 신규 테이블 4개 + 기존 테이블 필드 추가

### 1-0. 데이터 층위 — 소유자·대상물·건물 분리 (2026-07-14 확정)

"돈 내는 단위"와 "시설 있는 단위"를 층으로 분리한다.
```
소유자 = 고객(대상물, customers)  ──  소방안전관리비 지불 단위 (계약·과금·수금·점검·보고서)
   └ 건물(동, buildings) 1..N     ──  시설현황·층별수량 단위 (물리적, fire_facilities)
```
| 대상 | 단위 | 저장 |
|------|------|------|
| 소방안전관리비(월 정액)·점검료 | **고객(대상물)** 1건 | customers.monthly_fee_*/fee_* |
| 시설현황·층별수량 | **건물(동)** 마다 | fire_facilities.building_id |
| 점검·보고서·수금행(§4-7) | **대상물(고객) 1건**(다수동 통합) | inspections / customer |

→ 동을 나눠도 **과금·수금은 고객 1건으로 유지**(관리비는 건물이 아니라 고객에 붙음). 시설현황만 동별.

**✅ A+B 처리 확정 (2026-07-14) — 선택적 소유자 그룹**: 평소 A(대상물별 개별 청구), 가끔 B(소유자 통합청구)를
**한 스키마**로. 대부분은 비워두면 A로 동작, 통합 필요한 몇 건만 소유자로 묶으면 B.
- **`owners`(신규 테이블)**: 소유자명·연락처·biz_no(사업자면)·메모
- **`customers.owner_id`(nullable FK)**: null=독립 대상물(A, 기본·기존과 동일) / 같은 owner_id 공유=통합 그룹(B)
- **통합 청구(B)**: 세금계산서는 소유자가 단일 사업자면 통합 발행(각 대상물=품목 줄), 사업자 다르면 개별.
  안전관리 페이지(§4-7)에 **[소유자별 보기] 토글** → 소유자 소계·통합 청구/입금
- **통합 입금 배분**: 소유자 일괄 입금 → 입금문자 파싱(§4-6-1)이 소유자 합계와 매칭 → 미수에 배분
  (오래된 것부터 자동 or 수동 선택). 예: 132만 → 강순기건물 40만 + 상가 92만
- **나머지 무영향**: 소유자는 **청구·수금 편의 층**일 뿐 — 시설현황·점검·보고서·관계인은 대상물/건물 단위 그대로.

### 1-3. fire_facilities — 소방시설 현황 (신규) · **건물(동) 단위 확정 (2026-07-14 Q2)**

**건물(동) 단위(`building_id`)로 저장** — 다수동(§3-5)에서 동마다 설비가 다를 수 있으므로.

## 2. 데이터 이관 설계 (Phase 1)

### 2-1. 고객 일괄 등록 — `migrate-customers.mjs`

```
[읽기] 소방점검리스트.xls 26년 시트(+25년 옵션)
   ↓
[정규화] 상호 trim/공백 정리 · 지역 → region_myeon 매핑 · 전화번호 분리(여러 개면 첫번째=대표)
   ↓
[중복 판정] (상호, 지역) 키로 dedupe — 최신 연도 우선
   ↓
[매핑]  (2026년 시트만 — Q4)
   대상물명   → customers.customer_name
   지역       → region_si/myeon + fire_station 자동(지역→소방서 매핑 §1-1a)
   연면적     → buildings.total_area
   사용승인일 → customers.use_approval_date (형식 불량건은 리포트에 표기 후 null)
   점검계획일 → 26년 점검날짜(월/일) → plan_anchor_date (§6 확정)
   구분       → inspection_type (종합/작동, 외관기능·인수·일반 → 일반관리 §6-Q6)
   관계인     → customer_contacts (role=대표, name, phone)
   금액       → 종합/작동=monthly_fee_*, 일반=fee_* (과금유형 파생 §4-7)
   ↓
[시설현황 이관] 과거 작동보고서 엑셀의 현황 시트 → fire_facilities(건물 단위) 자동, 없으면 수동 보완(Q1)

[검수 리포트 출력] 등록 N건 / 중복 병합 M건 / 날짜 파싱 실패 K건 목록 → 콘솔+CSV

[실행 모드] --dry-run(기본, 리포트만) / --execute (스테이징) / 운영은 검수 승인 후에만
```

- **연간 점검계획 자동생성**은 등록 액션이 아닌 스크립트 직접 insert이므로, 이관 후 `POST /api/cron/generate-yearly-plans?year=2026` 1회 수동 발화로 일괄 생성.

### 3-0. 핵심 원리 — '개요' 시트 데이터 허브 (2026-07-14 확정)

실물 보고서(강순기·경복·곤빌딩, 30여 시트)의 구조: **모든 데이터는 '개요' 시트 1장에 모여 있고,
나머지 시트(갑지·정보·위임장·공문·계약서…)는 개요를 엑셀 수식으로 참조**해 자동 채워진다.
(현행 수기 방식도 개요에만 타이핑하면 나머지가 딸려 채워짐) — **✅ 2026-07-14 수식 참조 구조 확인(D)**.

→ **ERP는 '개요' 1장 + 독립 몇 장만 주입하면 되고, 30여 장을 각각 매핑하지 않는다.**
```
ERP가 개요 시트 데이터 주입 → Gotenberg(LibreOffice) 변환 시 수식 재계산 → 전 시트 자동 완성 → PDF
```
(exceljs는 수식 재계산을 안 하지만 LibreOffice 변환 단계에서 재계산되므로 성립)

**개요 → ERP 매핑**
| 개요 항목 | ERP 출처 |
|-----------|----------|
| 주된 점검인력 이름+경력수첩번호 | inspections.assigned_employee_id → profiles.name + `license_no`(P3-1) |
| 보조 점검인력(여러 명)+경력수첩번호 | **신규 `inspection_participants`**(점검별 참여자, role=보조) → profiles.license_no |
| 점검기간·점검일자 | inspection_start_date(1단계 확정일) + 점검일수 |
| 년도·발신일자·문서번호 | 생성 시점 + 문서번호 시퀀스 |
| 이행조치 필요기간 | 5·6단계 마감일 |
| 상호·소재지·용도·동수·세대수 | customers + buildings |
| 관할소방서 | 신규 필드 or 지역→관할서 매핑 |
| 관계인 이름·직위·생년월일·연락처 | customer_contacts + position·birth_date(P3-1) |

**시트별 역할** (개요 참조로 자동 작성)
- **보고서(갑지)**: 법정 표지 — 점검종류 체크(작동/종합/[√]최초=is_initial §3-5f)·대상물·점검기간·점검자(㈜승진소방=company_profile)·전자우편 동의
- **정보**: 대표자·소방안전관리자·등급·**소방계획서 작성/보관 여부(§8 fire_plans 보유로 자동 [√])**·자체점검·교육훈련
- **위임장**: 위임인=관계인(성명·생년월일·직위) / 수임인=㈜승진소방 (개요+company_profile)
- **공문(대상처)**: 관계인 통보 — 발신·수신·문서번호·발신일자·결과 통보문
- **계약서**: 갑=고객 / 을=승진소방 / 계약금액=점검료·정액(§4-7) / 계약기간(contract_date~)
- **현황·현1~5·세1~5·대상물**: 소방시설 현황·수량 (§3-3 fire_facilities)
- **소화기구·자탐1~2·유도등·기타**: 설비별 점검표 결과 ○/X (§4-5)
- **계획서(이행계획서)·완료보고서**: 불량(defects) + 이행조치 기간
- **목차**: 보유 설비 기준 자동 구성

**신규 데이터 소요**: ① 직원 `license_no`(P3-1) ② **`inspection_participants`(주된1+보조N)** ③ 관계인 position·birth_date(P3-1) ④ 관할소방서 필드.
누락 시 생성 전 체크리스트로 차단(§3-1 ②).

### 3-5. 다수동 처리 — '다수동일때' 시트의 "2. 건축물 정보" (2026-07-14 확정)

원칙: **1동 = buildings 1행** (기존 고객 1:N 건물 구조를 동 단위로 사용).

**(a) buildings 컬럼 추가 (migration 062)** — 양식 항목 중 미보유분
| 컬럼 | 소스 | 자동/수동 |
|------|------|----------|
| `permit_date` 건축허가일 | 건축물대장 API `pmsDay` | 자동 |
| `building_area` 건축면적 | API `archArea` | 자동 |
| `use_approval_date` 동별 사용승인일 | API `useAprDay` (동별) | 자동 — 고객 레벨 값은 주건축물 기준 유지(계획 로직 무영향) |
| `stairs` jsonb {직통:n, 피난:n} | API 미제공 | **수동** |
| `parking` jsonb {옥내지하,옥내지상,옥외,기계식} | API 일부(기계식) | **수동+일부자동** |

연면적·높이·층수·세대수·구조·지붕·승강기는 기존 컬럼(037/038) 그대로. 건물동수 = buildings 행 수 자동 계산.

**(b) 자동 수집 확장**: 현재 `fetchBuildingLedgerAction`은 연면적 최대 동 1개만 파싱 →
표제부 응답의 **전체 동(dongNm별 레코드)을 파싱해 buildings를 동별 생성**하도록 확장.

**(c) 입력 화면**: 고객 상세 [건물] 탭 — 동별 목록/편집. 수동 입력은 계단·주차장 2가지뿐.
보고서 누락 검증에 "B동 계단 정보 없음 → [건물 편집]" 포함.

**(d) 보고서 생성 분기**:
```
동 수 = 1 → '정보' 시트 단일 블록 ('다수동일때' 출력 제외)
동 수 ≥ 2 → '다수동일때' 시트: 템플릿에 5동분 블록 배치 → 동 순서대로 주입
            → 남는 블록 행 숨김(빈 블록 미출력) → 6동 이상은 시트 복제
체크박스(구조·지붕·주차): 저장값 → 해당 [ ]를 [√]로 텍스트 치환 (단일동 로직 재사용)
```

**(e) 구조·지붕 체크박스 매핑 규칙**: DB에는 건축물대장 원문 텍스트를 그대로 저장(정보 소실 방지)하고,
보고서 생성 시점에 포함 판정 매핑표(STRUCTURE_MAP/ROOF_MAP)로 양식 보기를 체크한다.
예: "철근콘크리트구조" → [√]콘크리트구조 / "철골철근콘크리트" → 콘크리트+철골 복수 체크 /
미분류 원문 → [√]기타( 원문 기입 ). 동이 여러 개면 동별로 각각 판정.

**(f) 최초점검 처리** (2026-07-14 확정): **최초점검 = 신축 고객의 생애 첫 "종합 1차"와 동일한 슬롯이다.**
- 신축 건물이 사용승인 후 60일 이내 받는 종합점검의 1회성 형태 = 그 고객의 첫 종합 1차 (이듬해부터의
  종합 1차는 "그 밖의 종합점검"). ERP가 종합 1차를 기준일(사용승인일) 월에 배치하므로 60일 요건도 자연 충족.
- 따라서 별도 유형·수동 입력 없이 **자동 판정**한다: 종합 1차 점검 생성 시 그 고객의 종합 이력
  (`inspections` + `legacy_inspections`)이 전무하면 `inspections.is_initial = true` 자동 세팅 (수동 해제 가능).
- 보고서 갑지 체크 분기: is_initial → `종합점검([√]최초점검)`, 아니면 `([√]그 밖의 종합점검)`.
- 이관 고객(기존 건물)은 과거 이력이 있으므로 자동으로 전부 '그 밖의 종합점검' — 오판 위험 없음.

### 3-6. 이행계획서(계획서)·이행완료보고서(완료보고서) 생성 (2026-07-14 확정)

불량 생애주기의 산출물. 계획서와 완료보고서는 **같은 불량 목록을 공유** — 계획서="고칠 예정",
완료보고서="고침 완료 + 증거".

**(a) 생애주기 (ERP 6단계 연동)**
```
⑤ 점검 X → 불량 등록(defects)+조치전 사진
 → 4단계: 이행계획서(계획 조치일, 소방서 제출)
 → 5단계: 소방보수(실제 공사)
 → 조치완료 입력(조치일·조치내용·조치후 사진)
 → 6단계: 이행완료보고서(완료보고서) 생성
```

**(b) 데이터 소스**
| 완료보고서 내용 | 출처 |
|-----------------|------|
| 표지(대상물·관계인·소방서·문서번호·완료일) | 개요 허브(§3-0) — 갑지와 공유 |
| 불량 목록(점검번호·위치·불량내용) | `defects` |
| 조치내용·조치완료일 | **기존 이행계획 `action_plans` 모듈 + 확장 필드** |
| 조치 전/후 사진 | 조치전=inspection-defects(기존) / **조치후=신규 업로드**(같은 버킷) |
| 이행조치 기간(기한) | 개요의 이행조치필요기간(5·6단계 마감일) |

**(c) 생성 흐름**: 점검 6단계 [완료보고서 생성] → 불량+조치정보 수집 → **누락 검증(미조치 불량·조치후
사진 없으면 차단 + 바로가기)** → 개요 허브 + '완료보고서' 시트 주입 → Gotenberg PDF → 자동 인쇄.

**(d) 특수 상황**
- **불량 0건**(점검표 전부 ○): 이행할 것 없음 → "지적사항 없음" 처리 또는 완료보고서 생략
- **기한 초과**: 조치완료일 > 이행조치 기간 → ⚠ 경고(제출은 가능, 담당자 확인)
- **부분 조치**: 전부 조치돼야 완료보고서 생성 — 일부만이면 계획서 상태 유지

**(e) 신규 데이터 소요**: ① `action_plans` 확장(조치완료일·조치내용) — 기존 모듈에 없으면 ② 조치 후 사진
업로드(inspection-defects 버킷 재사용, kind='after').

### 3-7. 다일 점검 — 점검기간 2~3일 처리 (2026-07-14 확정)

현재 `inspections.inspection_start_date`(단일)만 있어 기간 개념이 없음. 대형 건물의 2~3일 점검 지원.

**(a) 영향 4곳·처리**
| 영역 | 처리 |
|------|------|
| ① 점검일 확정 입력 | 단일 날짜 → **시작일 ~ 종료일**(기본 종료=시작 → 1일). 2~3일은 종료일 지정 |
| ② 6단계 마감일 기산점 ★ | 후속 절차는 점검 완료 후 시작 → 기산점 = **점검 종료일**. step2~6 = 종료일+N영업일. **1일이면 시작=종료라 기존과 동일(하위호환)**. 트리거·recalc(048/050)의 base_date를 종료일로 |
| ③ 보고서 개요 점검기간 | "시작 ~ 종료 (총 N일)" 자동 — 개요 허브(§3-0)에 시작·종료·일수 주입 |
| ④ 달력·모니터링 | 기간 막대 또는 시작일 "(2일)" 뱃지. **지연·초과 판정은 시작일 기준 유지** |

**(b) 데이터**: `inspections.inspection_end_date`(기본=시작일) + `inspection_days`(기본=종료−시작+1 자동,
**비연속 점검은 수동 override**). plan_item은 시작일(scheduled_date) 유지 + 종료일 미러/파생.

**(c) 비연속 점검**(예: 11/28+12/1): 일수 수동 입력(달력=실제 방문일, 개요="총 2일"). 기본 연속 가정.

**(d) 하위호환**: 1일 점검(대다수)은 시작=종료 → 기존 6단계 계산·달력·초과판정 100% 동일. 2~3일만 종료일 추가.

### 3-8. 직인·대표자 도장 자동 날인 (2026-07-14 확정)

보고서·공문·계약서 등의 날인란에 회사 직인·대표자 도장을 자동으로 얹는다.

**(a) 방식**: ★**엑셀 이미지 앵커**(권장) — 템플릿 날인 위치에 투명 PNG를 exceljs로 앵커 삽입 →
LibreOffice가 PDF 렌더. (기존 개요 허브 주입 파이프라인 재사용, 셀 고정으로 안정적) /
대안: PDF 생성 후 pdf-lib 좌표 오버레이(정밀하나 문서별 좌표맵 부담).

**(b) 날인 위치**
| 문서 | 회사 직인 | 대표자 도장 |
|------|:--:|:--:|
| 보고서 갑지(점검업체란) | ● | |
| 공문(대상처) 발신처 | ● | |
| 계약서 을(乙) | ● | ● |
| 위임장 수임인(승진소방) | ● | ● |
| 완료보고서 발신 | ● | |

- 회사 직인 = `company_profile.mark_url`(기존 필드) 재사용 / 대표자 도장 = 신규 `rep_seal_url`
- 둘 다 **배경 투명 PNG**. 대표자명은 기존 `company_profile.representative`.

**(c) 관계인 도장 제외**: 위임장 위임인·확인란은 고객 측 날인 → 자동 불가. 공란(실물 날인) 또는
고객 제공 이미지 있을 때만 옵션 날인.

**(d) 보안**(인감 도용 방지): 도장 이미지는 **비공개 스토리지 + 관리자만 접근**, 날인 생성 시 **사용 로그**,
**법인 인감 대신 사용인감 권장**, **문서별 자동 날인 on/off 토글**(민감 문서는 실물 날인만).

**(e) 법적**: 이미지 날인 실무 통용되나 **제출처 전자날인 인정 여부 확인 권장**. 미인정 시 "도장 없는 버전
인쇄 → 실물 날인"(엑셀 다운로드 버튼이 그 역할).

> - **현장 모드 (태블릿·휴대폰 브라우저)**: 반응형 + 큰 터치 타깃, 설비 1종씩 순차 진행
> - **사무실 모드 (데스크톱 ERP)**: 점검 상세 화면의 [점검표] 탭 — 전체 항목을 그리드로 한눈에,
>   키보드 입력(1=○, 2=X, 3=／, ↓다음 항목), **[전체 정상 ○]** 일괄 버튼 후 X 항목만 수정하는 방식.
>   현행 실무(현장 종이 수기 → 사무실 전산 입력)를 그대로 지원하는 것이 목적.
> - 두 모드는 같은 데이터에 쓰므로 현장에서 입력하다 사무실에서 이어서 완성하는 혼용도 가능.
>   입력·수정 이력은 점검 건에 기록(누가 언제).

### 4-5-1. 구체 예시 — 소화기구 및 자동소화장치 점검표 (2026-07-14)

- **시딩**: '소' 시트 → 그룹(1-A 소화기구/1-B 자동확산/1-C 자동소화장치) + 항목(1-A-001~) + 작동○/종합● 구분
- **표시 필터**: 작동점검이면 ●항목(예: 1-A-009 설치수량)이 아예 미표시, 종합이면 전부 표시
- **X 터치 → 불량 자동 연결**: 1-A-007에 X → 불량 폼이 점검번호·표준문구("소화기 충압 불량")를 자동으로
  채우고 위치·수량만 입력 (defect_catalog 코드 = 점검표 항목코드 동일 체계)
- **보고서 반영 3곳**: ① '소화기구' 점검표 면 — 항목별 ○/X/／ 주입 ② '현황·현1' 면 — 소화기 수량은
  점검표가 아닌 fire_facilities에서 (역할 분리) ③ 불량 요약·이행계획서 — X 항목이
  "01-A-007 소화기 충압 불량 (2층 복도 1개)" 형태로 목록화되어 관계인 통보·이행조치 근거가 됨
- **저장**: 점검 건별 응답 테이블(inspection_id × 항목코드 × ○/X/／ + 메모) — 현장/사무실 모드 공용,
  입력자·시각 이력 기록
- 흐름 한 줄: 현장 X 터치 → 불량 자동 등록 → 보고서 점검표 면 X 인쇄 + 불량 요약 인쇄 + 이행계획 연결
  (재입력·옮겨 적기 0회)

**점검표 노출 트리거** (2026-07-14 확정): 스위치는 **고객 상세 [소방시설 현황] 탭의 설비 체크박스**
(`fire_facilities.installed`) 하나다.
- [√] 소화기구 체크 → 점검 시작 시 [점검표] 탭 목록에 "소화기구 및 자동소화장치 점검표" 등장
  (미체크 설비의 점검표는 목록에 안 나옴 — 43종 중 해당 설비만)
- 하위 체크 → 그룹 단위 대응: [√]소화기=1-A 표시, [√]자동확산=1-B 표시, 미체크 장치의 그룹(1-C 등)은
  **자동 ／(해당없음) 처리** — 점검자가 일일이 ／ 누르지 않음
- 수량 필드(분말 12 등)는 점검표 노출과 무관 — 보고서 '현황·현1' 수량표 전용 (체크=노출, 수량=인쇄)
- 점검 유형이 두 번째 필터: 작동점검이면 ●항목 미표시
- **시설현황 미입력 고객**은 점검표 탭에 목록 대신 "먼저 시설현황 입력" 안내 + 바로가기 (43종 전체
  노출 방지 — 보고서 누락 검증과 동일 패턴)

**일반화 원칙** (2026-07-14, 자탐 문답 확정): 43종 점검표 전부 동일 메커니즘 —
`시설현황 체크(노출 스위치) → 점검유형 ○/● 필터 → 하위 미체크 그룹 자동 ／ → X=불량 자동 연결
→ 보고서 해당 면 주입`. 설비별 차이는 ① detail 구성(예: 자탐은 수신기 형식·회로수·감지기 종류별
수량 — fire_facility_floors 층별 그리드의 열이 곧 감지기 종류)과 ② 보고서 면 수(자탐은 자탐1·2 두 면)뿐.
**점검표 엔진은 1개, 43종은 시딩 데이터로 구동** — 설비별 화면 43개를 만드는 것이 아님.

### 4-6. 소방점검리스트 페이지 — 연간 통합 대장 (2026-07-14 추가 — 미구현, 지시 대기)

"올해 점검 + 수금 현황 한눈에 보기" 엑셀 대장을 앱 안으로 들여와, 이관·점검·청구가 한 화면에 모인다.

**(a) 위치·성격**: 좌측 메뉴 신규 항목 [점검 대장] (또는 점검 하위). **읽기 중심 + 계산서·입금 인라인 편집**.
연도 셀렉터로 과거·현재를 오간다.

**(b) 데이터 소스 — 연도로 분기**
| 연도 | 소스 | 비고 |
|------|------|------|
| 과거(ERP 도입 전, ~2025) | `legacy_inspections` (§1-2) | 이관된 원본 대장 — 읽기전용 |
| 현재·이후(2026~) | `inspections` + `inspection_plan_items` + 청구(billing) + `customers` | 실시간 집계 |

**(c) 컬럼 매핑** (엑셀 대장 ↔ ERP 데이터)
| 대장 컬럼 | ERP 출처 |
|-----------|----------|

**(e) 엑셀 내보내기**: 현행 대장 양식과 동일한 컬럼으로 xlsx 다운로드(exceljs) — 세무·백업·기존 습관 유지용.
과거 시트를 그대로 대체하지 않고 **병행**(이관 검수 및 회계 인수인계 목적).

**(f) 의존성·구현 메모**
- 금액·계산서·입금 컬럼은 **P4(청구 연동)**가 있어야 완전 동작 → 그 전에는 해당 열 공란/비활성.
- 과거 연도 표시는 **P1-B(legacy_inspections 이관)** 완료가 선행.
- 신규 페이지 1개(서버 컴포넌트 목록 + 인라인 편집 액션) — 별도 마이그레이션 불필요(기존·예정 테이블 조합).
- 점검확정 화면(월 단위 계획)과 역할 구분: 점검확정=진행 관리 / **소방점검리스트=연 단위 실적·수금 대장**.
- ✅ **P4 청구 방향 확정 (2026-07-14)**: 상세 청구·세금계산서는 **기존 청구 모듈**이 담당, 소방점검리스트
  대장은 **조회 + 계산서·입금 인라인 편집**만 (모듈을 대장에 흡수하지 않음). 대장의 인라인 편집은 청구 모듈에 반영.

### 4-6-1. 입금 확인 — 입금 알림 문자 붙여넣기 파싱 (2026-07-14 확정 방식 — 미구현, 지시 대기)

> ★ 채택 방식(2026-07-14): 은행 API 계약 없이, **이미 폰에 오는 은행 입금 알림 문자(카톡/SMS)를
> ERP에 붙여넣어 파싱**해서 청구 건과 매칭한다. 비용 0, 추가 인프라 0.

**(a) 동작 흐름**
```
① 은행 입금 알림 (예: "[농협] 08/14 14:22 입금 44,000원 이윤엽 잔액 1,234,567")
        ↓ 담당자가 문자 복사
② 대장 상단 [입금문자 붙여넣기] 창에 붙여넣기 (한 건 또는 하루치 여러 건 일괄)
        ↓ ERP 파싱 → 금액·입금자명·일시 추출
③ 미입금 청구 건과 매칭 후보 제시:
   "44,000원 이윤엽 → 수국빌딩 2026 청구? [입금확정][다른건][무시]"
        ↓ 사람이 확인 (자동확정 안 함 — 오매칭 방지)
④ 대장 입금일 자동 기입 + 미수금 감소
```

**(b) 파싱** — 은행마다 문자 형식이 달라 2단 처리:
- 1차: 은행별 정규식 템플릿(농협 등 주거래 은행)
- 실패 시: **Claude AI 파싱 폴백**(기보유 ANTHROPIC_API_KEY, §9 음성 파싱과 동일 원리) — 비정형 문자에서
  금액·입금자·일시 추출, 형식 변경에도 대응

**(c) 매칭**: 금액 + 입금자명 → 미입금 후보 제시. **예금주≠상호**(수국빌딩 입금자 "이윤엽") 문제는
`billing_autopay.holder_name`(§1-5) 별칭으로 해결. 확정은 사람이 클릭.

**(d) 보안**: 붙여넣은 문자에서 **금액·입금자·일시만 저장, 잔액은 버림**(원문 미보관/마스킹).

**(e) 더 자동화(후순위 선택)**: 안드로이드 문자 자동전달 앱 → 은행 알림만 서버 웹훅으로 전송하면 붙여넣기도
생략(완전 자동). 단 폰 상시 켜둠 필요·아이폰 제약. 1차는 붙여넣기 방식으로 충분.

**(f) 청구 안내 알림(선택)**: 카카오 알림톡(비즈니스 채널+발신대행사+템플릿승인)으로 청구 안내·입금 확인
통지. ERP 기존 SMS의 상위 업그레이드(실패 시 SMS 대체). 도입 시 건당 과금.

> 참고: 완전 자동 계좌연동(오픈뱅킹/쿠콘 CODEF)은 계약·요금이 필요해 **후순위**. 붙여넣기 파싱이 비용 대비
> 효과가 가장 크므로 1차 채택.

### 4-7. 안전관리 페이지 — 연간 청구·수금 관리 대장 (2026-07-14 추가 — 미구현, 지시 대기)

`안전관리 2026년.xlsx`(9시트)를 ERP로 재현. **소방점검리스트(§4-6, 점검 실적 축)의 형제 페이지로
수금 현황 축** — 고객 1곳 × 12개월 매트릭스. 밑단은 청구 모듈 데이터 공유, 보는 각도만 다름.

**(a) 위치·성격**: 좌측 메뉴 [안전관리 대장] (또는 청구·수금). 연도 셀렉터. 읽기 + 셀 클릭 상태전환.

**(b) 탭 구성** (엑셀 9시트 정리)

**(d) 데이터·마이그레이션**
- 청구 레코드에 **fee_type('monthly'|'inspection')** — 값은 **inspection_type에서 파생**(종합/작동→monthly,
  일반→inspection). (고객×연·월×type) 집계. 기존 청구 모듈에 월별·type 구조 없으면 확장.

## 7. 기존 시스템 연계 — 업무 전체 흐름도 (단계별 입력→처리→출력)

> 범례: 〔기존〕= 이미 운영 중인 모듈 그대로 사용 / 〔확장〕= 기존 모듈에 필드·기능 추가 / 〔신규〕= 이번에 새로 개발

### 7-0. 전체 흐름 한눈에

```
 ① 고객 등록          ② 연간 점검계획        ③ 점검일 확정          ④ 점검 시작
 〔기존+이관〕          〔기존 자동생성〕        〔기존〕                〔기존 6단계〕
 엑셀 26년 시트   ──▶  plan_anchor_date  ──▶  달력 드래그 /    ──▶  1단계 완료 =
 → customers           기준 연 12건 생성       클릭 확정              inspections +
 + 관계인/건물                                 (1~6단계 날짜 산출)     inspection_steps 생성
                                                                          │
 ⑧ 청구·수금           ⑦ 제출·완료 처리        ⑥ 보고서 생성·인쇄       ▼
 〔확장〕               〔기존〕                〔신규〕ⓅⓊⓈⒽ      ⑤ 현장 점검 입력
 fee_untaxed/taxed ◀── 3단계(관계인 제출)  ◀── [보고서 생성] 버튼 ◀── 〔신규 웹〕
 + billing_autopay      4단계(소방서 제출)      → PDF 자동 인쇄        점검표 ○/X/／

                                               → Storage 보관          + 시설현황 확인

### 7-1. 단계별 상세 — 입력 / 처리 / 출력

**① 고객 등록** 〔기존 화면 + 신규 이관 스크립트〕
| | 내용 |
|---|---|
| 입력 | (일괄) `소방점검리스트.xls` 26년 시트 → `migrate-customers.mjs` / (개별) 고객등록 화면 — 주소 검색 시 건축물대장 자동수집(사용승인일 포함, 기배포) |
| 처리 | `customers` + `customer_contacts` + `buildings` 생성. 점검계획일 = 26년 점검일의 월·일(확정 정책). 점검료는 `fee_untaxed/taxed`에 저장 |
| 출력 | 활성 고객 마스터 → ②의 입력. 과거 10년 이력은 `legacy_inspections`로 별도 적재(고객 상세 '과거 이력' 탭) |

**② 연간 점검계획 자동생성** 〔기존 — 수정 없음〕
| | 내용 |
|---|---|
| 입력 | ①의 customers (inspection_type 종합/작동 + plan_anchor_date) |
| 처리 | `generateYearlyPlanItems`: 기준일 우선순위(점검계획일→최초점검시작일→사용승인일)로 특별 1·2차 + 정기 = 연 12건 생성. 이관 직후 크론 수동 1회 발화 |
| 출력 | `inspection_plan_items` (planned) → 점검확정 화면·점검달력에 표시 |

**③ 점검일 확정** 〔기존 — 수정 없음〕
| | 내용 |
|---|---|
| 입력 | 관리자: 목록 클릭 확정 or 달력 정기 칩 드래그(기배포) |
| 처리 | `confirmPlanItemStageOneAction`: scheduled_date + step1~6 마감일 계산(영업일·공휴일 반영) |
| 출력 | confirmed 항목 → 모니터링 그리드·점검달력 동기화(기배포) |

**④ 점검 시작 = 6단계 생성** 〔기존 — 수정 없음〕
| | 내용 |
|---|---|
| 입력 | 담당자: [점검 시작] 버튼 (미배정이면 시작자가 자동 배정) |
| 처리 | `inspections` 생성 → 트리거가 `inspection_steps` 6단계 생성 → 확정일 기준 마감일 동기화 |
| 출력 | 1단계(점검일) 완료 → 2~6단계 재계산. 점검 상세 화면이 ⑤·⑥의 작업 허브가 됨 |

**⑤ 점검표 입력 — 현장·사무실 겸용** 〔신규 — 웹(반응형), §4-5〕 ★ P3-4
| | 내용 |
|---|---|
| 입력 | (현장 모드) 점검자가 태블릿/폰 브라우저에서 항목별 ○/X/／ 터치 · (사무실 모드) 데스크톱 ERP 점검 상세 [점검표] 탭에서 그리드 일괄 입력 — [전체 정상 ○] 후 X만 수정, 키보드 1/2/3 입력. **현장 종이 수기 → 사무실 전산 입력 방식 그대로 지원.** X 항목은 즉시 불량 등록(점검번호 자동완성 = `defect_catalog`) |
| 처리 | 두 모드가 같은 데이터에 저장(혼용 가능, 입력자·시각 이력 기록). 기존 inspection_sheets 구조 활용, 시딩된 43종 표준항목. 시설현황(`fire_facilities`)은 최초 1회 입력 후 매년 재사용 — 점검 때는 확인·수정만 |
| 출력 | 설비별 점검결과 + 불량 목록 → ⑥ 보고서의 점검표면·불량 데이터 소스 |

**⑥ 보고서 생성·자동 인쇄** 〔신규〕 ★ P3-1~3
| | 내용 |
|---|---|
| 입력 | 점검 상세에서 [보고서 생성 ▾ → 작동점검 보고서] 클릭 (입력할 게 없음 — 데이터는 ①~⑤에서 이미 쌓임) |
| 처리 | (1) 수집: 고객·건물·관계인(직위)·점검인력(경력수첩번호)·기간·시설현황·점검표결과·불량 → (2) 누락 검증(부족하면 체크리스트+바로가기) → (3) exceljs 템플릿 주입 → (4) Gotenberg PDF 변환 → (5) Storage 저장 + `generated_reports` 기록 |
| 출력 | **새 탭 PDF + 인쇄 대화상자 자동 표시** / 엑셀 다운로드(전자민원 제출용) / 생성 이력에서 언제든 재인쇄 |

**⑦ 제출·완료 처리** 〔기존 — 수정 없음〕
| | 내용 |
|---|---|
| 입력 | 담당자: ⑥의 출력물을 관계인 제출(3단계)·소방서 제출(4단계) 후 단계 완료 체크 |
| 처리 | `completeStepAction` → 모니터링 그리드·달력 D-day 자동 갱신(기존) |
| 출력 | 6단계 완주 → 점검 completed → 이듬해 ②로 순환 |

**⑧ 청구·수금** 〔확장〕 ★ P4
| | 내용 |
|---|---|
| 입력 | ①에서 저장된 점검료(fee_untaxed/taxed) + 자동이체 정보(`billing_autopay`, 계좌 암호화) |
| 처리 | 기존 청구 모듈로 청구서 발행·입금 처리 — `안전관리 2026.xlsx` 수기 대장을 대체 |
| 출력 | 세금계산서 발행 관리·미수금 현황 (기존 화면) |

### 7-2. 보고서 데이터 흐름도 — 어떤 테이블이 보고서 어느 면을 채우나

```
customers ─────────────┬─▶ [보고서 갑지]  상호·주소·용도·점검구분
buildings ─────────────┤─▶ [건축물 정보]  연면적·층수·구조·승강기·사용승인일
customer_contacts ─────┤─▶ [정보·공문·위임장] 관계인 성명·직위(확장)·전화·생년월일(확장)
profiles ──────────────┤─▶ [개요·배치]   점검인력 성명·경력수첩번호(확장)
inspections+steps ─────┤─▶ [갑지·개요]   점검기간·이행조치 기간(단계 마감일)
fire_facilities(신규) ─┤─▶ [현황 현1~5]  설비 [√]체크·수량 상세
fire_facility_floors ──┤─▶ [층별 집계]   층×설비 수량표
점검표 응답(시딩 43종) ─┤─▶ [설비별 점검표] 항목별 ○/X/／
defects+defect_catalog ┴─▶ [불량내용]    점검번호별 표준 불량 문구
                            │
                            ▼
             exceljs 주입 → Gotenberg PDF → 인쇄/보관
```

핵심: **⑥에서 새로 입력하는 것은 없다.** 보고서는 ①~⑤ 과정에서 자연스럽게 쌓인 데이터의 "출력"일 뿐이며,
부족한 데이터는 생성 시점에 체크리스트로 알려주고 해당 입력 화면으로 바로 보낸다.

② 업로드:   ERP 고객 상세 [소방계획서] 영역에 HWP 원본 + PDF(한글 "PDF로 저장") 두 벌 업로드
             — 연도별 버전 이력 보관 (Supabase Storage)
③ 재인쇄:   ERP [소방계획서 인쇄] 버튼 → 저장된 PDF 새 탭 + 인쇄 대화상자 자동
             (점검 보고서와 동일한 인쇄 UX — 단 자동 '생성'이 아니라 업로드본 인쇄)
④ 선택 확장: [계획서 데이터 내려받기] 버튼 — 고객·건물·관계인 데이터를 CSV로 제공,
             한글 메일머지(누름틀 필드)로 기본 정보 자동 채움 (반자동, 서버 개발 불필요)
```

요약: ERP가 HWP를 직접 열거나 인쇄하지는 못한다(웹 기술 한계). 최초 작성만 한글에서 하고,
PDF를 올려두면 이후 인쇄는 ERP 버튼 한 번이다.

### 8-2. 지도·사진 부속자료 설계 (2026-07-14 추가 → ✅ 구현됨: 부속자료 업로드 + 생성 본문 사진 삽입)

**(a) DB** — `fire_plan_attachments` (migration 057에 포함 예정)
| 컬럼 | 설명 |

**사진 저장 규약** (2026-07-14 확정 제안):
- 경로: `fire-plans/{고객ID}/{연도}/attachments/{timestamp}_{kind}(.jpg|_thumb.jpg)` — 파일은 스토리지,

  + 300px 썸네일 생성 + EXIF 회전 보정 + **EXIF(GPS 등) 자동 소거**(재인코딩 부수효과, 보안 이점).
  리사이즈 후 장당 2MB 상한. HEIC 미지원 → JPEG 공유 안내
- 삭제 시 본 이미지+썸네일 스토리지까지 함께 제거 (고아 파일 방지)
- 용량 전망: 고객 100곳 × 4장 ≈ 연 210MB — 무료 플랜(1GB) 4~5년 여유 (원본 저장 시 1년 내 초과라 리사이즈 필수)

**(b) UI** — 소방계획서 행 펼침 → 부속자료 썸네일 그리드
```
▼ 2026년 소방계획서 (개정 1)                    [인쇄 ▾] [PDF] [HWP]
   부속자료 (4)                                        [+ 추가]

```

**(c) 인쇄 — 본문+부속 병합** : [인쇄 ▾] 드롭다운
```
· 본문만 인쇄          → 기존 방식 (업로드된 표준양식 PDF)
· 전체 인쇄(본문+부속) → 서버에서 pdf-lib로 병합: 본문 PDF 뒤에 부속자료를

```

**(d) 지도(위치도) 저장 방식** (2026-07-14 확정 제안): **최종 저장물은 이미지(JPEG)로 통일** —
kind='map'으로 사진과 동일 파이프라인(2000px+썸네일, 같은 버킷)을 사용해 인프라 추가 없음.
제출 시점에 고정된 그림이어야 하므로 좌표+동적 지도 저장은 하지 않는다. 생성 경로 2가지:
- **1차(업로드)**: 약도 스캔·캡처 이미지를 종류='위치도'로 업로드 — 즉시 가능, 실무 충분
- **2차(캡처 생성기, 선택 확장)**: [위치도 자동 생성] → Leaflet+OpenStreetMap 지도 모달(키 불필요,

  타일 canvas 캡처가 보안 정책에 걸릴 수 있어 차선)
- 배치도(layout)·피난경로도(evacuation)는 스캔본이 PDF인 경우가 많아 **PDF 그대로 저장 허용** —
  병합 인쇄 시 페이지째 결합.

### 8-3. 연차 제출 방안 — 변경분 자동 주입 발행 (2026-07-14 확정 제안)

매년 대부분 동일 + 일부만 변경(연도·작성일·관계인·자체점검 일정·시설현황)되는 문서의 반복 제출 대응.

| 방안 | 방식 | 자동화 | 공수 |
|------|------|-------|------|
| A. 한글 메일머지 | 누름틀 필드 + ERP CSV → 한글에서 병합 | 중 | 소 |
| **B. HWPX 표시자 치환 (권장)** | 양식을 HWPX(공개 XML)로 저장, `{{고객명}}` 표시자 → ERP가 데이터 주입한 완성 HWPX 생성 | 높음 | 중 |
| C. 웹 재현 완전 자동 | 양식 전체 HTML 재구축 → PDF | 완전 | 대 (B 운영 후 재평가) |

**B안 운영 흐름**: [최초 1회] 표준양식 HWPX화+표시자 삽입 → ERP에 발행 템플릿 등록 →
[매년] 고객 상세 [연차발행] → ① ERP가 최신 데이터(연도·관계인·**연간 점검계획의 자체점검
예정일**·시설현황) 주입한 HWPX 생성·다운로드 → ② 한글에서 검토·서술부만 수정 → PDF 저장 →
**소방서 제출은 이 시점에 바로 가능(인쇄 또는 전자민원)** → ③ ERP 업로드(revision 기록) →
④ 제출 처리(제출일·관할서 기록). 사람 작업이 "옮겨 적기"에서 "검토"로 축소, 점검 일정은 ERP 계획과 항상 일치.

**업로드의 의미** (2026-07-14 문답 확정): 문서는 두 상태로 존재한다 — ERP가 내려주는 것은 "초안"이고,
한글에서 검토·수정된 "최종본"은 담당자 PC에만 생긴다 (편집이 ERP 밖에서 일어나므로).
**업로드는 제출 요건이 아니라 최종본을 회사 보관함에 넣는 마무리 단계**다 — 이것이 있어야
재인쇄·개정이력·내년 발행 시작점·미제출 배지가 작동하고, 담당자 PC 교체·퇴사에도 유실이 없다.
제출 먼저, 업로드 나중 순서도 무방.
**✅ 2026-07-14 확정: 업로드 단계는 없애지 않고 유지한다** — 최종 제출본의 회사 보관이 필수 요건.
(C안을 도입하더라도 "최종본이 ERP 보관함에 남는다"는 원칙은 동일하게 유지)

**ERP 다운로드 3종**: ① 주입된 HWPX 초안(검토 시작점) ② 최종 PDF(재인쇄·재제출·관계인 송부)
③ HWP/HWPX 원본(다음 개정·내년 발행의 수정 시작점).

**제출 추적**: `fire_plans.submitted_at`(제출일) + `submitted_to`(관할 소방서) 추가(057에 포함) →
"올해 미발행·미제출 고객" 배지(§8-1(d)) + 연초 알림(선택).

**도입 순서**: ① 056 기구현분으로 수동 운영 시작 → ② B안(템플릿 표시자 반나절 + ERP 주입 기능) →
③ C안은 B 운영 경험 후 재평가.

> 구현 순서(→ ✅ 구현됨, migration 086): revision·attachments 마이그레이션 → 보관함 UI 확장(연차발행·개정판·부속자료)
> → 병합 인쇄 → B안 HWPX 주입. 기구현분(056: 기본 업로드·자동 인쇄)은 그대로 하위 호환 — 기존 행은 전부 revision 1로 간주.

## 9. 작동점검결과 음성 문서화 (2026-07-14 제안 — 미구현, 지시 대기)

현장에서 말로 남긴 점검 결과를 ERP에 텍스트로 문서화하고 불량 등록까지 연결한다.

```
[현장] 점검 화면 🎤 녹음 ─▶ ① STT(음성→텍스트) ─▶ ② AI 구조화(Claude) ─▶ [검토·확정]
       오디오 원본 보관        실시간 받아쓰기        불량 후보+점검번호 매핑     불량 일괄 등록
       (증빙·재변환용)                                                           점검 비고 저장 → 보고서 반영
```

**단계별 도입**
1. **실시간 받아쓰기** — Web Speech API(크롬 내장, 한국어, 무료). 점검 상세/점검표 화면 [🎤 음성 입력],
   말하면 텍스트 실시간 축적 + 오디오 원본(webm, 분당 ~0.5MB) 스토리지 보관. 소음 오탈자 전제 → "초안" 취급
2. **AI 구조화** — 받아쓰기 텍스트를 Claude API(기보유 ANTHROPIC_API_KEY)로 구조화:
   불량 사전(defect_catalog 93건)을 참조로 주고 "2층 소화기 압력 불량 → 01-A-007" 식 후보 제안.
   **AI는 제안만, 등록은 사람이 체크·확정** (법정 문서 — 자동 확정 금지)
3. **서버 STT 승격(선택)** — 정확도 부족 시 클로바 스피치(유료) 또는 VPS whisper.cpp(무료·느림)로 승격.
   원본 오디오를 보관하므로 과거 녹음 재변환 가능

**데이터 모델** — `inspection_voice_notes` (신규): inspection_id FK · audio_path · transcript ·
structured(jsonb, AI 제안) · status(draft→confirmed) · created_by/at.
마이페이지 음성 메모(개인용)와 별개 — 점검 건에 귀속되어 불량·보고서와 연결되는 것이 차이.

**흐름 연계**: §7-1 ⑤(점검표 입력)의 보조 입력 수단 — 받아쓰기로 잡힌 불량이 defect_catalog 자동완성과
같은 경로로 등록되고, ⑥ 보고서의 불량내용·비고에 반영된다.
