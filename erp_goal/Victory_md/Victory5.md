# 이행계획/완료 제출현황 모니터링 — 화면 설계 v1

> 점검결과의 불량내역을 기반으로 이행계획 제출대상을 조회하여 이행계획을 자동으로 생성하고 제출 여부를 모니터링합니다.  
> 이행계획에 기반하여 이행완료보고서를 자동으로 생성하고 제출 여부를 모니터링합니다.

---

## 1. 화면 개요

| 항목 | 내용 |
|------|------|
| 경로 | `/action-plans/status` |
| 접근 권한 | manager / admin |
| 구성 | 상단: **이행계획 제출현황 모니터링** / 하단: **이행완료보고서 제출현황 모니터링** (한 페이지, 두 섹션) |
| 데이터 기준 | 점검결과 불량내역이 1건 이상인 점검건 자동 대상 포함 |

---

## 2. 이행계획 제출현황 모니터링 (상단)

### 2-1. 필터 영역

```
┌─────────────────────────────────────────────────────────────────────────┐
│  [제출대기 ▼]  계출일 [2025-03-28] ~ [2025-04-18]  [🔍검색]            │
│                                            건물명 [▼]  [검색어  ]  [🔍] │
└─────────────────────────────────────────────────────────────────────────┘
```

| 필터 | 옵션 | 설명 |
|------|------|------|
| 상태 드롭다운 | 전체 / 작성대기 / 제출대기 / 제출완료 | 이행계획 작성·제출 진행 상태 |
| 계출일 범위 | DATE ~ DATE | 이행계획 제출기한 기준 기간 |
| 건물명 | 텍스트 검색 | `customers.customer_name` ILIKE |

---

### 2-2. 테이블 구성

```
┌────┬──────────────┬──────────────┬────────────┬────────────┬──────────┬──────┬──────────────┬──────────────────┬──────────────┬────────────────────┐

├────┼──────────────┼──────────────┼────────────┼────────────┼──────────┼──────┼──────────────┼──────────────────┼──────────────┼────────────────────┤
│  1 │ 맥시티       │ 종합정밀점검 │ 2022-06-25 │ 2024-03-23 │ —        │ —    │ —            │ 3                │ —            │ —                  │
└────┴──────────────┴──────────────┴────────────┴────────────┴──────────┴──────┴──────────────┴──────────────────┴──────────────┴────────────────────┘
```

### 2-3. 컬럼 상세 정의

| 컬럼 | 설명 | 데이터 소스 |
|------|------|-------------|
| No | 행 번호 | — |
| 건물명 | 고객(건물)명 | `customers.customer_name` |
| 구분 | 점검 유형 | `inspection_plan_items.inspection_type` |
| 점검완료일 | 실제 점검 완료일 | `inspection_report_status.inspection_completed_at` |
| 보고일 | 점검보고서 발행일 | `inspection_reports.issue_date` |
| 계출기한 | 이행계획 제출 마감일 | 보고일 기준 자동 산출 (보고일 + 30일) |
| 송부일 | 관계인 송부일 | `action_plan_status.sent_at` |
| 소방서제출일 | 소방서 이행계획 제출일 | `action_plan_status.submitted_at` |
| 증명서내역(갯수) | 불량 증명서류 수량 | `inspection_defects` COUNT |
| 이행계획제출일 | 이행계획서 실제 제출일 | `action_plans.submitted_at` |
| 이행완료보고제출일 | 이행완료보고서 제출일 | `action_complete_reports.submitted_at` |

---

### 2-4. 상태별 화면 표시

| 상태 | 조건 | 표시 |
|------|------|------|
| 작성대기 | 불량내역 있음 & 이행계획 미작성 | 행 배경 연노랑 |
| 제출대기 | 이행계획 작성 완료 & `이행계획제출일 IS NULL` | 행 배경 연주황 |
| 제출완료 | `이행계획제출일 IS NOT NULL` | 정상 표시 |

---

### 2-5. 하단 범례

> ★ **작성대기**: 불량내역에 대한 이행계획이 작성되지 않은 건  
> ★ **제출대기**: 이행계획보고서가 제출되지 않은 건 (이행계획제출일이 없는 건)

---

### 2-6. 이행계획 자동 생성 플로우

```
점검완료
    │
    ▼
불량내역 확인 (inspection_defects)
    │
    ├── 불량 없음 → 이행계획 불필요, 목록 미표시
    │
    └── 불량 1건 이상 → 이행계획 제출 대상 자동 등록
              │
              ▼
        이행계획 자동 생성 버튼 활성화
              │
              ▼
        이행계획서 PDF 자동 생성
        (불량내역 + 조치계획 포함)
              │
              ▼
        소방서 제출 → 이행계획제출일 기록
```

## 4. DB 테이블 설계

### 4-1. inspection_defects (점검 불량내역)

> 점검 완료 후 등록된 불량 항목. 이행계획 자동 생성의 원본 데이터.

| 컬럼 | 타입 | 비고 |
|------|------|------|
| id | UUID PK | |
| inspection_id | UUID → inspections(id) CASCADE | |
| defect_code | VARCHAR(20) | 불량 항목 코드 |
| defect_name | VARCHAR(200) | 불량 설비명 |
| defect_detail | TEXT | 불량 내용 상세 |
| photo_url | TEXT | 불량 사진 URL |
| severity | VARCHAR(10) | 경중 (경미 / 보통 / 중대) |
| created_at | TIMESTAMPTZ | |

---

### 4-2. action_plans (이행계획서)

> 불량내역 기반으로 자동 생성되는 이행계획서.

| 컬럼 | 타입 | 비고 |
|------|------|------|
| id | UUID PK | |
| inspection_id | UUID → inspections(id) CASCADE | |
| plan_file_url | TEXT | 자동 생성된 이행계획서 PDF 경로 |
| completion_target_date | DATE | 불량 조치 완료 목표일 |
| submitted_at | DATE | 소방서 제출일 |
| sent_at | DATE | 관계인 송부일 |
| created_by | UUID → profiles(id) | |
| created_at / updated_at | TIMESTAMPTZ | |

---

### 4-3. action_complete_reports (이행완료보고서)

> 불량 조치 완료 후 생성되는 이행완료보고서.

| 컬럼 | 타입 | 비고 |
|------|------|------|
| id | UUID PK | |
| action_plan_id | UUID → action_plans(id) CASCADE | |
| report_file_url | TEXT | 자동 생성된 이행완료보고서 PDF 경로 |
| completed_at | DATE | 불량 조치 완료일 |
| submitted_at | DATE | 소방서 제출일 |
| created_by | UUID → profiles(id) | |
| created_at / updated_at | TIMESTAMPTZ | |

---

### 4-4. action_plan_status (이행계획 진행 상태 로그)

> 이행계획 단계별 날짜 기록 (상단 테이블 데이터 소스).

| 컬럼 | 타입 | 비고 |
|------|------|------|
| id | UUID PK | |
| action_plan_id | UUID → action_plans(id) CASCADE | |
| sent_at | DATE | 관계인 송부일 |
| fire_station_submitted_at | DATE | 소방서 제출일 |
| defect_certificate_count | INT DEFAULT 0 | 증명서내역 수량 |
| updated_by | UUID → profiles(id) | |
| created_at / updated_at | TIMESTAMPTZ | |

## 5. 비즈니스 규칙

| 규칙 | 내용 |
|------|------|
| 자동 대상 포함 | `inspection_defects` 1건 이상 존재 시 이행계획 제출 대상 자동 등록 |
| 이행계획 자동 생성 | 불량내역 조회 후 버튼 클릭 1회로 PDF 자동 생성 |
| 이행완료보고 자동 생성 | 이행계획 제출 완료 후, 조치 완료 결과 입력 시 PDF 자동 생성 |
| 기한 초과 강조 | `오늘 > 계출기한` AND 제출일 없음 → 해당 셀 `bg-red-100 text-red-700` |
| 제출완료 시 해제 | 제출일 기록 완료 시 강조 자동 해제 |
| 페이지네이션 | 각 테이블 독립 페이지네이션, 기본 10행/페이지 |

## 6. 상태 흐름도

```
[점검완료]
    │
    ▼
[불량내역 등록]
    │
    ├─ 없음 → 이행계획 불필요 (화면에 미표시)
    │
    └─ 있음 ─→ [작성대기] ──→ (이행계획 자동 생성)
                                    │
                                    ▼
                              [제출대기] ──→ (소방서 제출)
                                    │
                                    ▼
                              [이행계획 제출완료]
                                    │
                                    ▼
                              [조치 완료 입력]
                                    │
                                    ▼
                    (이행완료보고서 자동 생성)
                                    │
                                    ▼
                              [이행완료 제출대기] ──→ (소방서 제출)
                                    │
                                    ▼
                              [이행완료보고 제출완료]
```
