# 점검계획등록 — 신규 화면 설계

> 전월 점검계획 + 근무달력 기반 **월간 점검계획 자동 생성** 모듈.
> 기존 `/inspections/new`(단건 배정)와 별개로, 한 달치 전체 고객의 점검 일정을 한 번에 수립·관리한다.

---

## 1. 화면 목록

| 경로 | 설명 | 권한 |
|------|------|------|
| `/inspection-plans` | 월간 점검계획 목록 (달력 + 리스트) | 전 직원 |
| `/inspection-plans/new` | 계획 수동 등록 폼 | manager/admin |
| `/inspection-plans/[year]/[month]` | 특정 월 계획 상세·수정 | manager/admin |
| `/inspection-plans/[year]/[month]/auto` | 자동 생성 미리보기 & 확정 | manager/admin |

## 6. DB 설계

### inspection_plans (월간 점검계획 헤더)

| 컬럼 | 타입 | 비고 |
|------|------|------|
| id | UUID PK | |
| year | SMALLINT | |
| month | SMALLINT | 1–12 |
| status | TEXT | draft / confirmed / in_progress / completed |
| auto_generated | BOOLEAN | 자동 생성 여부 |
| ref_plan_id | UUID → inspection_plans(id) | 참고한 이전 계획 |
| created_by | UUID → profiles(id) | |
| confirmed_at | TIMESTAMPTZ | 확정 일시 |
| created_at / updated_at | TIMESTAMPTZ | |

`UNIQUE(year, month)` — 월별 1개 계획

### inspection_plan_items (계획 항목)

| 컬럼 | 타입 | 비고 |
|------|------|------|
| id | UUID PK | |
| plan_id | UUID → inspection_plans(id) CASCADE | |
| customer_id | UUID → customers(id) | |
| inspection_type | inspection_type | 고객 기본값 or 오버라이드 |
| sequence_num | SMALLINT | 1 or 2 |
| scheduled_date | DATE | 점검 예정일 |
| assigned_employee_id | UUID → profiles(id) | |
| contact_id | UUID → customer_contacts(id) | 선택 |
| status | TEXT | planned / confirmed / completed / cancelled |
| inspection_id | UUID → inspections(id) | 실제 점검 생성 후 연결 |
| notes | TEXT | |
| created_at / updated_at | TIMESTAMPTZ | |

`UNIQUE(plan_id, customer_id, sequence_num)` — 계획 내 동일 고객·차수 중복 방지

## 7. 상태 흐름

```
[계획 수립]
     │
     ▼
  draft ──────────────────────────────────┐
  (자동생성 미리보기 / 수동 등록 중)       │
     │                                  [삭제]
  [확정]
     │
     ▼
confirmed
  (점검 예정일 도래 시 자동 → in_progress)
     │
     ▼
in_progress
  (전체 항목 완료 시 자동 → completed)
     │
     ▼
completed
```

**plan_items 상태 흐름:**

```
planned → confirmed → completed
              │
           cancelled (담당자 변경·일정 취소)
```

**inspections 연동:**

계획 항목의 `scheduled_date`가 되면 manager/admin이 **"점검 시작"** 버튼으로
→ `inspections` 테이블에 실제 점검 레코드 생성
→ `plan_items.inspection_id` 에 연결
→ 기존 7단계 워크플로우 자동 시작

---

## 8. 주요 비즈니스 규칙

| 규칙 | 내용 |
|------|------|
| 월별 1개 계획 | `UNIQUE(year, month)` — 같은 달 계획 중복 불가 |
| 종합 고객 연 2회 | 1차·2차 모두 같은 월 배치 불가 (최소 30일 간격 권장) |
| 공휴일 배정 | 허용하되 인라인 경고 표시 |
| 자동 생성 기준 | 전월 계획의 담당직원·일정 간격 패턴 재사용 |
| 계획 → 실제 점검 | `plan_items.status = confirmed` 항목만 점검 시작 가능 |
| 계획 취소 | `inspection_id`가 이미 연결된 항목은 취소 불가 |

### 10-5. 빨간 강조(Red Cell) 규칙

| 강조 대상 셀 | 강조 조건 | 의미 |
|-------------|----------|------|
| 배치신고예정일 | `today > 배치신고예정일` AND `배치신고일 IS NULL` | 배치 신고 기한 초과 — 즉시 처리 필요 |
| 제출기한 | `today > 제출기한` AND `fire_station_submitted = false` | 소방서 제출 기한 초과 |
| 송부일 | 배치신고일 기록 후 n일 이내 송부 미완료 | 송부 지연 |

> 하단 범례:  
> ★ 점검보고서가 소방서에 미제출(반납)된 건 중 날짜가 입력되지 않은 건  
> ★ 점검완료일과 보고서 제출기한 사이 기준: ① 관계인 통보 ② 소방시설관리자 확인 ③ 소방서 제출  
> ★ 보고서 날짜가 기준일과 일치하면 보고서 생성 버튼 활성화
