## 3. AUTH — 인증·권한·세션

#### AUTH-1. 정상 로그인·로그아웃

- **단계**: ① `/login`에서 admin 계정 로그인 ② 대시보드 표시 확인 ③ 헤더에서 로그아웃
- **기대**: 로그인 → `/dashboard` 이동. 로그아웃 → `/login` 복귀, 뒤로가기로 대시보드 재진입 불가

#### AUTH-2. 로그인 실패 잠금 (5회/30분)

- **단계**: park.dev 계정으로 틀린 비밀번호 5회 입력
- **기대**: 1~4회 "이메일 또는 비밀번호가 올바르지 않습니다. (n/5회)" 카운트 표시 → 5회째 잠금 안내 → 올바른 비밀번호로도 "계정이 잠겨 있습니다. N분 후" 표시. `activity_logs`에 `login_failed` 기록. (해제는 admin이 DB에서 `locked_until` null 처리 또는 30분 대기)

#### AUTH-3. 비활성 계정 차단

- **사전**: admin이 임시 직원 1명 비활성화 (ADM-2와 연계)
- **기대**: 해당 계정 로그인 시 "비활성화된 계정입니다" — 비밀번호가 맞아도 차단

#### AUTH-4. 미인증 접근 차단 (미들웨어)

- **단계**: 로그아웃 상태에서 `/dashboard`, `/customers`, `/admin/users` URL 직접 입력
- **기대**: 전부 `/login` 리다이렉트. `/login`, `/api/cron/*`만 통과

#### AUTH-5. 역할 승격 차단 (B안 매트릭스 기준)

- **단계**: ① employee(park.dev)로 `/admin/users`, `/approvals`, `/accounting/vouchers`, `/billing/status`, `/customers/regional-assign` 직접 접근 ② manager(dev.manager)로 `/admin/users` 접근
- **기대**: 전부 차단(리다이렉트). 반대로 employee가 `/customers/new`, `/action-plans`, `/inspections/new`, `/buildings/new`는 **정상 진입** (B안 개방 확인, §1-4 매트릭스 전 행 확인)

#### AUTH-7. 서버 액션 권한 재검증 (B안 경계)

- **단계**: employee 세션으로 매니저 전용 서버 액션 직접 호출 — 고객 삭제(`deleteCustomerAction`), 담당 배정(`assignEmployeeAction`/`bulkAssignEmployeeAction`), 점검 삭제(`deleteInspectionAction`), 인라인 담당자 변경(`patchCustomerFieldAction`의 `assigned_employee_id` 필드)
- **기대**: UI 노출 여부와 무관하게 서버에서 거부. 반대로 고객 등록·계획 확정·이행계획 액션은 employee 세션에서 성공

#### AUTH-6. 로그인 성공 시 실패 카운터 리셋

- **단계**: 3회 실패 후 올바른 비밀번호로 로그인 → 로그아웃 → 다시 1회 실패
- **기대**: 실패 카운트가 (1/5회)부터 재시작 (누적 4회 아님). `activity_logs`에 `login_success` 기록

## 4. DASH — 대시보드

#### DASH-1. KPI 카드 정합성

- **사전**: 시드 데이터 투입 완료
- **단계**: admin 로그인 → 4개 KPI(담당 고객/진행 점검/기한 초과/이달 완료) 숫자 확인 → 각 원천 화면(고객관리·점검업무)의 실제 건수와 대조
- **기대**: KPI = 원천 목록 건수 일치

#### DASH-2. 역할별 대시보드 범위

- **단계**: 같은 화면을 park.dev(employee)로 재확인
- **기대**: employee는 **자기 담당 건만** 집계 (admin 수치와 다름), 마감 임박 목록도 담당 건 한정

#### DASH-3. D-Day 색상 규칙

- **사전**: 마감 초과 1건·3일 이내 1건·7일 이내 1건이 존재하도록 점검 단계 날짜 조정
- **기대**: 🔴 기한 초과 / 🟡 3일 이내 / 🟢 7일 이내 색상 매핑 정확. 사이드바 뱃지(빨강=지연·D-Day, 주황=D-1~3) 수치도 동일 기준으로 일치

#### DASH-4. 사이드바 회사 브랜딩 (2026-07-08 변경)

- **단계**: ① 사이드바 로고 텍스트 확인 ② `/company`에서 업체명을 `테스트상호`로 변경 저장 → 새로고침 ③ 원복(`승진소방ENG`)
- **기대**: 로고 텍스트가 company_profile 업체명을 따라 변경됨. 로고 이미지 등록 시 기본 보라 아이콘 대신 이미지 표시

- **단계**: 연락처 등록(이름·전화·이메일·소속) → 이름/전화로 검색 → 삭제
- **기대**: 검색 부분일치 동작, 개인 격리 확인

#### MY-4. 쪽지함 `/my/messages`

- **단계**: park.dev → jung.dev로 쪽지 발송 → jung.dev 로그인, 받은쪽지 확인·열람 → park.dev 보낸쪽지에서 읽음 표시 확인

#### MY-5. 노트 `/my/notes`

- **단계**: 마크다운(제목·굵게·목록) 포함 노트 저장 → 렌더링 확인
- **기대**: 마크다운 정상 렌더, 개인 격리

#### MY-6. 녹음 메모장 `/my/voice-memos`

- **단계**: 마이크 권한 허용 → 5초 녹음 → 저장 → 목록에서 재생

#### MY-7. 결재서명 `/my/signature`

- **단계**: 서명 패드로 직접 서명 저장 → 이미지 업로드로 교체

## 6. DOC — 전자결재

#### DOC-1. 기안 → 승인 풀사이클

- **단계**: ① park.dev가 문서 작성(유형: 보고서, 결재라인: dev.manager) → 기안 ② dev.manager 로그인 → 결재함에서 문서 열람 → 승인
- **기대**: 기안 시 결재자에게 알림(종 아이콘). 승인 후 상태 `승인`, 기안자에게 알림. 결재자 서명(MY-7) 표시

#### DOC-2. 반려 → 재기안

- **단계**: 기안 → dev.manager가 **사유 없이 반려 시도** → 사유 입력 후 반려 → park.dev가 수정 후 재기안

#### STOCK-1. 품목·분류 등록 → 발주 → 입고

- **단계**: ① 분류(소화기) + 품목(ABC소화기 3.3kg, 최소재고 5) 등록 ② 거래처로 발주 10개 ③ `/stock/in`에서 발주건 선택 입고 10
- **기대**: 재고현황 = 10. 발주 상태 입고완료 전환

#### STOCK-2. 출고 → 재고 차감

- **단계**: `/stock/out`에서 6개 출고(용도: 점검, 담당 park.dev)
- **기대**: 재고 = 4 → **최소재고(5) 미달 빨간 표시**

#### STOCK-3. 재고 조정

- **단계**: 실사 결과 3개로 조정(사유 입력)
- **기대**: 재고 = 3, 조정 이력에 차이(−1)·사유 기록

#### STOCK-4. 재고 경계·동시성

- **단계**: ① 재고(3)보다 많은 5개 출고 시도 ② 두 브라우저에서 동시에 2개씩 출고
- **기대**: ① 음수 재고 차단(또는 경고 — 현행 동작 기록) ② 최종 재고가 정확히 반영되고 음수·유령 차감 없음

## 18. EX — 전체 시스템 예외(Exception) 테스트 시나리오

> 소방안전관리 모듈의 예외는 `Victory_test.md` §7(EX-A~H)이 담당한다. 여기서는 **그 외 전 모듈**의 예외를 같은 분류 체계로 커버한다.
> 각 케이스 공통 기대: **명확한 한국어 오류 안내 + 데이터 무변경 + 콘솔 크래시 없음**. 서버가 500을 반환하거나 화면이 빈 채로 멈추면 FAIL.

### EX-V. 입력값 검증

#### EX-V1. 회계 금액 경계

- **단계**: 전표 등록에서 ① 음수 금액 ② 0원 ③ 문자("abc") ④ 소수점 3자리 ⑤ 10억 초과 금액 입력
- **기대**: ①~③ 저장 차단 + 안내. ④⑤ 처리 규칙(반올림/차단) 일관 — 현행 동작 기록

#### EX-V2. 휴가 날짜 역전·과거 신청

- **단계**: ① 종료일 < 시작일 ② 이미 지난 날짜 ③ 같은 기간 중복 신청
- **기대**: ① 차단 ② ③ 정책대로(차단 또는 경고) — 중복 기간은 이중 차감으로 이어지면 안 됨

#### EX-V3. 재고·수량 경계

- **단계**: 입고/출고/조정에서 ① 음수 수량 ② 0 ③ 소수(단위가 개수인 품목) ④ 재고 초과 출고(STOCK-4 연계)
- **기대**: 전부 차단 또는 정의된 동작. 저장 후 재고현황 값 왜곡 없음

#### EX-V4. 견적·급여 계산 필드

- 신규 상태 기능 추가 시 §19 매트릭스와 INV 목록을 함께 갱신 (예: B안 권한 개편 → INV-E10 추가됨)

## 21. 테스트 결과 기록 양식

> `Victory_test_result.md`와 동일 규칙. 결과 파일: `Victory_entire_result.md` 권장.

| ID | 시나리오 | 결과 | 확인 내용 / 실패 상세 | 테스터 | 일시 |
|----|----------|------|----------------------|--------|------|
| AUTH-1 | 정상 로그인·로그아웃 | PASS/FAIL/BLOCKED/INFO | | | |

**기록 규칙**

- **PASS**: 기대 결과와 정확히 일치 (스크린샷 권장)
- **FAIL**: 불일치 — 재현 단계·실제 결과·기대 결과를 반드시 기록
- **BLOCKED**: 선행 케이스 실패·환경 문제로 수행 불가 — 원인 케이스 ID 기록
- **INFO**: 명세 없는 현행 동작 관찰 기록 (개선 후보)
- 실패 건은 `victory_test_result_fixing.json`에 FIX/IMP 항목으로 등재하여 추적한다
- 데이터 정리: 테스트 종료 후 생성한 테스트 고객·직원·전표·게시글은 비활성화 또는 삭제하고, ADM-5에서 변경한 회사 정보는 원복한다
