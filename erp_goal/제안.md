# 설정 페이지(/settings) 신설 제안 (2026-07-10)

> **처리 현황: 1차(1단계 + 3단계) 구현 완료 ✅ (2026-07-10)**
> - `/settings` 페이지 신설 — 내 정보 카드, 비밀번호 변경(현재 비밀번호 재검증), 결재 서명 바로가기,
>   관리 바로가기 6종(admin 전용) — 죽은 링크(404) 해소
> - 검증: 브라우저 자동 테스트 9건 통과 (`scripts/_verify-settings.mjs`) — 페이지 렌더·내 정보 표시,
>   잘못된 현재 비밀번호 거부, 새 비밀번호 불일치 거부, 변경 성공 후 새 비밀번호 로그인 확인
> - **2단계(알림 수신 설정) 구현 완료 ✅ (2026-07-10)** — 마이그레이션 051(profiles.notification_prefs),
>   공통 notify 헬퍼(lib/notify.ts), 발송 지점 8곳 교체(결재·휴가 결과 / 담당 배정 4곳 / 마감 크론),
>   설정 페이지 알림 카드(필수 2종 잠금 + 토글 4종). 검증: 실제 UI E2E — 담당 배정 알림 끄고
>   담당 이관 → 미수신, 켜고 이관 → 수신 확인 (scripts/_verify-notify-prefs.mjs)

사이드바 하단 "설정" 메뉴가 미구현 페이지(404)로 연결되는 죽은 링크임이 확인됨 (수정사항리스트 7번).
링크 제거 대신 설정 허브 페이지를 신설하는 안.

## 배경 — 현재 상태

- **본인 비밀번호 변경 기능이 없음** — 관리자에게 초기화(resetPasswordAction)를 요청해야만 변경 가능
- 자기 정보(사번·부서·직책·입사일)를 한눈에 확인할 화면이 없음
- 설정 성격 기능들이 흩어져 있음: 회사 정보 → /company, 직원·부서·공휴일·건물용도·활동로그 → 관리자 메뉴,
  결재서명 → My Page
- 알림(notifications) 시스템은 있으나 수신 여부 설정이 없음

## 제안 기능

### 1단계 — 개인 설정 (전 직원, 권장 필수)

| # | 기능 | 근거 | 규모 |
|---|---|---|---|
| 1 | **내 정보 카드** — 이름·사번·부서·직책·입사일·이메일 (읽기 전용, 수정은 관리자 문의 안내) | 자기 정보 확인 화면 부재 | 소 |
| 2 | **비밀번호 변경** — 현재 비밀번호 확인 후 새 비밀번호 설정 | 기능 자체가 없음. 보안 관행상 필수 | 소 |
| 3 | **결재 서명 바로가기** — 기존 /my/signature 링크 카드 | 설정 성격인데 My Page에 묻혀 있음 | 소 |

### 2단계 — 알림 수신 설정 (상세안, 2026-07-10 구체화)

**현재 알림 시스템 실태 (조사 결과)**

- 알림은 전부 **인앱 알림**(상단 종 아이콘, notifications 테이블) — 이메일(Resend)·웹푸시는 코드에 미구현
- 발송 지점이 5개 모듈 + 크론에 흩어져 각자 insert: 결재(documents), 휴가(leaves),
  점검 배정(inspections·customers), 점검 마감 임박(inspection-deadline-notify 크론)
- 수신자가 알림을 끄거나 고를 방법이 전혀 없음

**제안 — 카테고리별 수신 토글 4종**

| 카테고리 | 포함 알림 | 기본값 | 끄기 허용 |
|---|---|---|---|
| 결재 | 결재 요청 도착·승인·반려 | 켜짐 | **요청 도착은 끄기 불가(업무 필수)**, 승인·반려 결과만 선택 |
| 휴가 | 휴가 신청(승인자에게)·승인·반려 | 켜짐 | 결재와 동일 원칙 |
| 담당 배정 | 고객 담당 배정·점검 배정·인수인계 | 켜짐 | 허용 |
| 마감 임박 | 점검 단계 마감 임박(매일 00:05 크론) | 켜짐 | 허용 |

**구현 설계**

1. **DB**: `profiles.notification_prefs JSONB DEFAULT '{}'` 컬럼 1개 (마이그레이션)
   — `{ "approval_result": true, "leave_result": true, "assignment": true, "deadline": true }`,
   미설정 키는 켜짐으로 간주(기본 전체 수신). 채널이 늘면 `{ assignment: { inapp: true, email: false } }`로 확장 가능
2. **공통 발송 헬퍼**: `notify(admin, recipientId, category, payload)` — insert 전에 수신 설정 검사.
   흩어진 발송 지점 ~10곳을 헬퍼로 통일 (산재 코드 정리 효과 겸함)
3. **UI**: 설정 페이지에 "알림 설정" 카드 — 토글 4개 + 저장, 필수 항목은 잠금 표시
4. **크론**: 마감 임박 발송 시 수신자별 deadline 설정 필터
5. **검증**: 브라우저 테스트 — 토글 끄기 → 해당 알림 발생시켜 미수신 확인, 필수 알림은 항상 수신

**규모**: 중소 (마이그레이션 1 + 헬퍼 + 발송 지점 교체 + 설정 UI + 테스트)
**제외 범위**: 이메일·웹푸시 채널 신설 — 인프라(Resend 키·PWA 푸시) 세팅이 필요한 별도 프로젝트
