## 도메인·HTTPS (배포 전 결정, 2026-07-08)

> ✅ **확정: `sjfire.co.kr`** — Caddy 리버스 프록시 구성 완료(`erp/deploy/Caddyfile` + compose caddy 서비스). DNS A 레코드만 연결하면 HTTPS 인증서는 자동 발급된다.
> 결론: **엄밀히는 필수 아니지만, 실사용이면 사실상 필요.** 연 1~2만 원 도메인 구입 + Let's Encrypt 무료 HTTPS 구성을 권장.

| 배포 방식 | 도메인 없이 | 권장 |
|-----------|------------|------|
| Vercel (프리뷰) | `프로젝트명.vercel.app` 무료 서브도메인 + HTTPS 자동 — **그대로 사용 가능** | 프리뷰는 기본 도메인으로 충분 |
| VPS + Docker (프로덕션) | `http://공인IP:3000` 접속은 가능하나 비권장 | **도메인 + HTTPS 필수 수준** |

VPS에서 도메인이 사실상 필요한 이유:
1. **HTTPS**: Let's Encrypt 무료 인증서는 도메인 필요. HTTP만 쓰면 브라우저 경고 + 로그인 쿠키 보안 저하
2. **모바일 앱 erpUrl**: 고정 주소 필요 — IP 변경 시 앱 재배포
3. **직원 사용성**: IP:포트는 기억·공유 부적합
4. 비용 회피가 우선이면 DuckDNS 등 무료 서브도메인 + Let's Encrypt 조합도 가능 (회사 시스템 인상엔 유료 도메인 권장)

절차: 도메인 구입(예: sjfire.co.kr) → DNS A레코드를 VPS IP로 → Nginx/Caddy 리버스 프록시 + Let's Encrypt(certbot 또는 Caddy 자동) → 모바일 `erpUrl`·환경변수의 URL을 도메인으로 통일.

## ④ ERP(Next.js) 배포

### 🎯 Docker vs Vercel — 이 앱의 권장 선택

> **결론: 프로덕션은 Docker(자체 호스팅). Vercel은 프리뷰/스테이징 용도로만 선택적 사용.**
> 한 앱을 두 곳에 동시 프로덕션으로 띄우는 것은 세션·cron 중복 때문에 권장하지 않습니다.

**이 프로젝트가 Docker에 맞는 근거 (실제 설정 기반)**

| 근거 | 내용 |
|------|------|
| `output: 'standalone'` + `Dockerfile` 완비 | `Dockerfile`이 `.next/standalone` + `node server.js` 방식으로 이미 완성(멀티스테이지, non-root `nextjs` 유저). **Vercel은 `output: standalone`을 무시**하고 자체 빌드를 쓰므로 이 설정이 무용지물이 됨. |
| 🚨 Server Actions `bodySizeLimit: '20mb'` | **결정적 제약.** Vercel 서버리스 함수는 요청 본문이 **4.5MB로 제한**되어, 점검사진·문서·서명·음성메모(최대 20MB) 업로드가 **Vercel에선 실패**함. 우회하려면 전부 Supabase Storage 직접 업로드로 재작업 필요. Docker엔 이 제한이 없음. |
| Cron (`CRON_SECRET`) | Docker는 시스템 cron/스케줄러 컨테이너로 자유롭게 운용. Vercel Cron은 플랜별 실행 빈도 제한이 있음. |
| 데이터 소재지·비용 | 국내 리전 VPS에 두기 쉽고, 실행당 과금 없이 예측 가능한 비용. |

**"둘 다" 쓴다면 역할 분리만 권장**
- **Docker = 실제 프로덕션** (무제한 업로드, cron, 자체 서버)
- **Vercel = 프리뷰/스테이징** (PR별 미리보기 URL, 화면·흐름 확인용)
- ⚠️ 단, Vercel 프리뷰에선 위 **20MB 업로드 제약**이 그대로 적용되므로 대용량 업로드 검증은 Docker 쪽에서 해야 함.

### 옵션 A — VPS + Docker (권장, 프로덕션)

`Dockerfile`과 `docker-compose.prod.yml`이 이미 저장소에 준비되어 있습니다. **인라인으로 다시 작성할 필요 없이 그대로 사용**합니다.

**1) 환경변수 파일 준비** — `F:\AI\ERP\erp\.env.production` 생성

```
NEXT_PUBLIC_SUPABASE_URL=https://ryuozdhnilfjlahorizh.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJ...
SUPABASE_SERVICE_ROLE_KEY=eyJ...
ANTHROPIC_API_KEY=sk-ant-api03-...
CRON_SECRET=<임의의 긴 난수>
BUILDING_LEDGER_API_KEY=<data.go.kr 발급키>
# (선택) RESEND_API_KEY, VAPID 키 등
```

**2) 빌드 & 기동** (compose 사용 — 권장)

```bash
cd erp
docker compose -f docker-compose.prod.yml up -d --build
# → 3000 포트로 자동 기동, restart: always 로 재부팅 시 자동 복구
```

또는 compose 없이 직접:

```bash
cd erp
docker build -t sjfire-erp .
docker run -d -p 3000:3000 --env-file .env.production --restart always --name sjfire-erp sjfire-erp
```

**3) HTTPS** — nginx 리버스 프록시 + Let's Encrypt(certbot)로 TLS 적용 권장.
`next.config.ts`의 HSTS 헤더는 `NODE_ENV=production`에서만 적용되므로 HTTPS 종단 필수.

> 참고: 로컬 검증 시 `next start`는 `output: standalone`에서 경고를 냅니다. 프로덕션은 위 Docker(`node server.js`) 방식이 정석입니다.

### 옵션 B — Vercel (프리뷰/스테이징 전용, 선택)

> ⚠️ **프로덕션 단독 사용은 비권장** — 20MB 업로드가 Vercel 4.5MB 본문 제한에 걸립니다.
> 화면·흐름 확인용 프리뷰로만 사용하세요.

1. https://vercel.com 가입 (GitHub 계정으로 로그인)
2. **New Project** → GitHub 저장소 연결
3. **Root Directory**: `erp` 입력
4. **Environment Variables**에 아래 입력:
   - `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `ANTHROPIC_API_KEY`
   - Cron 사용 시 `CRON_SECRET`, 건축물대장 연동 시 `BUILDING_LEDGER_API_KEY`
   - (선택) `RESEND_API_KEY`, VAPID 키
5. **Deploy** → `https://your-app.vercel.app` 자동 발급
6. PR마다 프리뷰 URL 자동 생성 (스테이징 검토용)

| 장점 | 한계 |
|------|------|
| 자동 HTTPS, PR별 프리뷰 | **요청 본문 4.5MB 제한** → 대용량 업로드 실패 |
| GitHub push 시 자동 재배포 | `output: standalone` 무시됨 / Cron 빈도 제한 |

### 옵션 C — Railway (월 ~$5, 간편 대안)

Docker 없이 Git만으로 배포. VPS 운영이 부담스러울 때의 절충안.

1. https://railway.app 가입 → **New Project** → **Deploy from GitHub**
2. 저장소 선택, Root Directory: `erp`

4. 자동 빌드·배포 → Settings → Domains에서 커스텀 도메인 연결
   - 업로드 한도·요금은 플랜에 따라 다르므로 20MB 업로드가 필요하면 사전 확인.
